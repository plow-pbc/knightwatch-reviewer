# Coding Standards

Canonical coding principles for my projects. Referenced from `~/.claude/CLAUDE.md` (all coding work) and `~/.claude/REVIEW_PRACTICES.md` (automated + human review). Hand-curated — auto-tuning from `learn-from-replies.sh` targets `COMMENT_REVIEW_MISTAKES.md` only, not this file.

## Team Context

Small team (~3 engineers), ~10 active users, moving fast. Design bias:

- **Elegance > defensive sprawl.** Code that reads cleanly and scales beats code that handles every hypothetical failure mode.
- **Concise code that fails loudly > verbose, defensive code with brittle special cases.** We prefer short code that captures the *spirit* of the spec or desired UX and crashes when its assumptions break, over long code that bakes in handling for every edge case the author imagined. Each special case calcifies a decision and invites the next one. When the spec seems to require contortions, the right reflex is to question the spec (see "Reframe the Spec" below), not to add branches.
- **Engineer time > compute time.** When picking between approaches, prefer the one that scales in *engineers' time* — fewer special cases, fewer conditional branches, fewer files to touch when the next variant lands. A solution that costs more compute or runs slower is fine if it eliminates a class of code we'd otherwise have to maintain. Lookup tables, hand-coded heuristics, and per-case branches are brittle in a way the bill-of-materials never reflects.
- **Loud breaks.** If an assumption is wrong, the code crashes — it does not silently degrade. No swallowed errors, no fallback chains, no softened tests.
- **Architecture decisions should not accrue tech debt.** A "fine-for-now" choice that the roadmap will force us to unwind in six weeks is a medium-severity finding, not a nit — flag it and propose the seam.
- **DRY and missing-abstraction findings are first-class.** They are not style polish. A missing abstraction today is tomorrow's bug surface × N.

Reviewers — human and automated — weight findings with these values in mind, not just raw severity.

## Fail-Fast / Offensive Programming

Write fail-fast code. If an assumption is wrong, the code crashes — it does not silently degrade.

- Access data directly: `x["key"]`, not `(x or {}).get("key", default)`
- No fallback chains: `a or b or c or default` — pick one source of truth
- No swallowed errors: `try: ... except: pass`
- Avoid `try/except` / `try/catch` in core logic; catch only at system boundaries (user input, external APIs)
- When encountering existing defensive code, rip it out and run tests. If they pass, the defense was bloat.

**Review question:** *Is this code non-defensive? Does it crash loudly when its invariants are violated?*

## Concise Code (LOC is a cost)

Every line earns its place. Verbose code is more bugs, more review time, more drift.

- Don't add `if x:` guards when `x` should always be truthy
- Don't introduce helpers/abstractions for one or two call sites
- Three similar lines > premature abstraction
- When I propose verbose code, push back
- Prefer direct access over fallback chains and nested conditionals
- Every conditional is a maintenance burden — all things equal, avoid it. Each `if`/`else` branch is a state the next reader has to hold and the next change has to update. When you find yourself adding a special case, ask whether a different seam would let you delete it instead.

**Review question:** *Is this concise and elegant, or is there extra code that doesn't pay for itself?*

## Anti-Bloat — Don't Propose These Remedies

LOC is a stand-in for **conditionals + special cases + defensive branches + new abstractions** — those are the actual cost being managed. Each calcified branch survives every future refactor and shapes the next change. The test for any edge-case handler: *does the edge case actually happen, or will it in the near future?* If neither, the remedy is bloat regardless of how the line count nets out.

When reviewing a PR or applying a bot's suggestion, **don't propose** (or accept) these patterns:

- **assert→raise+log replacements outside trust boundaries.** `Fail-Fast` prefers the assertion crashing loudly on internal-caller bugs.
- **isinstance / type-validation checks for internal callers.** Validation belongs at trust boundaries (user input, external APIs); internal calls trust their callers.
- **state-reset / fallback writes** for hypothetical pollution ("set X = default in case prior init left it dirty"). Unless the polluting scenario is observed in production, the fallback locks the surrounding code into preserving both branches forever.
- **wrapper dataclasses, snapshot views, or new DI seams** for one call site. These are remedies for repeated problems; "Three similar lines > premature abstraction" applies.
- **streaming / incremental rewrites** of small in-memory operations on theoretical perf or OOM grounds. Cite a measured failure or skip.
- **companion tests, CI guards, or regression coverage for unreachable scenarios** ("no test for prod re-enable", "no test that this can't happen"). Tests calcify too — a test for a scenario that doesn't occur preserves a contract that may have been wrong, and the next refactor has to keep that contract working forever.

When pushing back on a bot finding whose remedy hits one of these patterns, cite this section by name (`Anti-Bloat`) and apply the LOC-negative or branch-negative version of the fix.

**Review question:** *Does the proposed remedy add a branch / handler / abstraction for a scenario that hasn't been shown to actually occur?*

## Incremental Improvement

Leave each file a little better than you found it. "Better" = fewer LOC, fewer conditionals, fewer special cases, clearer assumptions at the seams it consumes and exposes, and louder failure when those assumptions break. A bit slower at runtime is fine. If you're already editing something and spot an obvious tidy — two nearly-identical mocks that could collapse into a factory, an unused import, a one-line comment that restates the function name — take it.

Split the work: get the business-logic change working and tests passing first, then do the cleanup as a **separate commit**. A reviewer should see the behavioral change cleanly on its own and the cleanup as its own diff.

- Don't block on unrelated cleanup — keep the focused change landable.
- Don't sneak cleanup into the feature commit — it hides the real change.
- If the cleanup is big enough that it deserves its own PR, make it its own PR.

**Review question:** *Did the author leave the file measurably better than they found it, without muddying the business-logic diff?*

## DRY: Find Existing Logic Before Adding New

New functions, user-facing strings, and status reports must be grep'd for near-duplicates before approval. Parallel implementations are the single most common LLM defect.

- Grep the repo for similar function names and near-duplicate strings.
- If a similar implementation exists, unify rather than adding alongside.
- Duplicate logic with an existing repo site: *blocking: dry* — cite the existing site by path.
- Intra-PR duplication (N near-identical blocks in the same diff) is a missing abstraction: *medium: dry*.

**Review question:** *Is this DRY with the rest of the codebase? Are we re-using existing functions where we should?*

## Regression Risk

New code must not quietly change existing business logic. Any change to a function's contract, return shape, side effects, or error behavior needs an explicit callout.

- Changed semantics of an existing function without updating call sites: *blocking: regression*.
- New conditional branch that alters legacy-path behavior: *medium: regression* unless proven safe.
- A bug fix without a regression test that exercises the old-bug path is *blocking: tests*.

**Review question:** *Does this introduce regressions? How does it change existing business logic, and is that change intentional and covered?*

## Reframe the Spec

Any single new block over ~20 LOC is a smell. Pause and ask whether a spec or UX tweak would delete most of that code. Don't silently accept contorted code to match a literal ask — name the simpler alternative so the author can choose.

- If satisfying a spec detail is adding notable complexity: *low: spec-reframe* with the alternative described in one sentence.
- If the complexity is structural (not just a spec ask): escalate to *medium*.

**Review question:** *Would a small change to the spec or UX accomplish the spirit of the request while significantly reducing complexity?*

## Name the Shape

Before writing code, name the *class* of problem: parsing, validation, dispatch, retry, auth, serialization, formatting, batching, state, idempotency, audit logging, feature-flag gating. Each is a recurring shape with a preferred home in the codebase. Pick the shape first; the lines come second.

Two failure modes:

- **Wrong shape.** Regex when the upstream could emit structured data. Hand-rolled validation when pydantic/zod is already in use. `if kind == "A": ... elif kind == "B": ...` instead of a dispatch dict. Bool-soup state instead of an enum. The code works on today's input and silently breaks on the next variant. Regex is the canonical example — treat regex on string-typed input as a smell that *structure got discarded upstream*. The fix is usually to make upstream emit data, not to grow the regex.
- **Five shapes for five similar problems.** Each new feature lands with its own auth check, its own retry loop, its own date format, its own error shape. Each PR looks fine in review; the policy now lives in 17 places. If you are the second instance of a missing shape, *that* is the moment to introduce one — not the fifth.

Before writing: grep for how the codebase already handles this class. If a shape exists, conform. If none exists and you are not the first instance, you have just discovered tech debt — flag it.

**Review question:** *Did the author name the problem class and conform to (or establish) the codebase's shape for it, or did they solve only the literal symptom?*

## Generalize the Fix (Narrow-Fix)

A fix tied to one reported input (regex, string equality, hard-coded special case) is a narrow fix. Ask: what is the next variant of this input, and does this code cover it?

- Narrow fixes that won't survive the obvious next bug are *medium: narrow-fix*.
- If the root cause is systemic (parsing, state machine, schema), fix the class, not the instance.
- `Narrow-Fix` is only valid on the FIRST occurrence of a class on a given PR. A second instance of the same class on the same PR escalates to `Bug-Class-Recurrence` (below) — repeating `Narrow-Fix` review after review erodes the tag's signal and traps the author in a local-fix loop.

**Review question:** *Is this a root-cause fix or a patch for one symptom?*

## Bug-Class-Recurrence

When the same class of bug has been flagged 2+ times on the same PR — across reviews, or across multiple findings within a single review — the right finding is structural, not local. Patching individual instances has reached diminishing returns; the right move is to name the architectural shape that would make the entire class impossible (per-session value types, sealed enums for state, single-owner data, registries, dispatch maps) and recommend that.

A `Bug-Class-Recurrence` finding **replaces** — does not append to — the individual local findings of the same class. Listing both anchors the author on the local fix; they will fix the local one and the structural finding becomes background noise.

If you genuinely cannot name the structural alternative, downgrade to `medium` and surface as an Open Question instead. Do NOT fall back to listing local fixes.

- Same class flagged in 2+ prior reviews → *blocking: bug-class-recurrence*.
- 2+ findings of the same class in the current review draft → *blocking: bug-class-recurrence* (collapse them into one structural finding).
- Supersedes `Narrow-Fix` after the first occurrence of the class.

**Review question:** *Is this the Nth instance of a recurring shape? If so, what's the structural alternative — the one piece of code that would make every instance disappear?*

## Tests

Tests must fail loudly when something is broken. Never add graceful degradation, skips, or soft failures to tests. A broken test should crash, not pass silently.

- Test **user experience and business logic**, not self-evident language/runtime behavior
- A change crossing multiple layers (e.g. Swift → plowd → API) needs 1–2 focused behavior tests, not exhaustive layer-by-layer coverage
- Prefer shared fixtures/factories over inline setup blocks
- Refactor repeated test payloads, DB rows, and lifecycle setup into helpers
- Remove low-value helper tests before adding more boilerplate
- **Run `just test` after every change** — features, fixes, refactors, anything. Don't return to the user with a red bar.
- **`just test` must pass before merging.** The pre-merge gate is non-negotiable: no merge on a red bar, even if the change is "obviously" safe. If a repo doesn't have a `just test`, add one before landing the first PR that needs gating.

New tests in the plow repo should align with `docs/TEST_PATTERNS.md` and `./tests/README.md` — the repo's own test-writing conventions. Tests that diverge from those patterns without a good reason are a *low: tests* finding.

**Review question:** *Are the tests loud-on-failure, aligned with the repo's test patterns, and focused on behavior rather than implementation details?*

## Migrations

Never hand-write Alembic migrations. Always use `alembic revision --autogenerate` to generate them from model changes.

**Review question:** *Was this migration autogenerated?*

## How reviewers should weight these

For any PR, reviewers rank findings by severity first (`blocking` → `medium` → `low` → `nit`), then within a severity band by:

1. **Tech-debt and architectural findings** (missing abstraction, DRY violation, design that won't survive the roadmap) — these compound.
2. **Broad-correctness findings** that affect many paths or users.
3. **Surface-area findings** touching many files.
4. **Localized fixes and style** rank LAST within their band.

If two findings compete and one is "code that won't scale as the team grows" vs. one that is "line-level style," the scalability finding wins the higher slot even at the same severity.

# Plow — Product Context

**Stage:** Early-stage agent platform, moving fast. Small user base; rigor should match that — favor clean designs over hardening edge cases.

**What it is:** A general wrapper that gives sandboxed agents narrow, permission-gated access to cloud services and on-device capabilities. Three tiers: **Plow API** (`api/`, FastAPI on AWS) + **Phoenix macOS app** (`app/Phoenix/` Swift + `app/plowd/` Python daemon, runs an Alpine VM with the agent runtime inside) + **Plow CLI** (`cli/`, the agent's tool surface). The current agent runtime is OpenClaw, but the wrapper is designed to be runtime-agnostic. Connectors and channels are pluggable and the inventory shifts — read `README.md` for the current list. ClawHub (`clawhub.ai`) is a *separate* service that hosts agent skills; the marketplace UI in this repo calls its API.

**Authoritative guidance lives in two files; read them first:**
- `AGENTS.md` — coding stance (clean replacement over migration code, no dual-write paths, no compatibility shims, complexity-debt aversion, scheduler-selection rules, issue-filing conventions for `cncorp/plow`).
- `docs/architecture/file-taxonomy.md` — every file is exactly one of (a) **upstream agent runtime** (currently OpenClaw, pulled in at VM build time; we have no edit access to that source — fixes there mean pinning a different version or PRing upstream, not editing in this tree), (b) **Plow wrapper code we own**, or (c) **per-install user state**. Confusing the categories is "the source of nearly every 'broken on upgrade' or 'shipped user data' incident." A pre-commit guard (`scripts/check_protected_paths.py`) enforces it.

**Architectural commitments worth flagging when a PR breaks them:**
- The agent never holds raw OAuth tokens — it sees narrow CLI tools only. PRs that leak token material into the agent surface are a security regression.
- `plowd` is the host/guest control plane. Code that talks to the VM directly (e.g. SSH from a service that already has a plowd dependency) splits authority — route through plowd instead.
- File-taxonomy violations are blocking. The two recurring failure modes: (i) per-install (c) state shipped inside `Plow.app/Contents/Resources/`, (ii) attempts to "fix" upstream (a) behavior by patching around it in this tree. `applyEnabled()` conflates “user preference” with “current assertion state.” When `IOPMAssertionCreateWithName` returns any non-success, the code flips `isEnabled` to `false` and immediately overwrites `UserDefaults` with that value. import Foundation
import IOKit.pwr_mgt
import Observation

/// Holds `kIOPMAssertionTypePreventSystemSleep` while the user has opted in, so
/// the system stays in Dark Wake and the OpenClaw VM keeps serving through
/// display sleep. On battery the kernel effectively downgrades the assertion to
/// idle-sleep-prevention, so a single assertion works in both power states;
/// draining the battery is the user's explicit choice when they toggle this on.
/// Clamshell sleep on a MacBook without an external display is still enforced
/// by the kernel and isn't reachable from a public API.
@Observable
@MainActor
final class KeepMacAwake {
    static let preferencesKey = "keepMacAwakeWhileRunning"

    private var assertionID: IOPMAssertionID = 0
    private var hasAssertion = false
    /// Set true while we're rewriting `isEnabled` from inside `applyEnabled` to
    /// revert a failed acquire — the recursive `didSet` reads this and bails.
    private var suppressApply = false

    var isEnabled: Bool = UserDefaults.standard.bool(forKey: KeepMacAwake.preferencesKey) {
        didSet { applyEnabled() }
    }

    init() {
        applyEnabled()
    }

    /// Releases the assertion deterministically before the app exits, so the
    /// system can coalesce any pending sleep state at logout/shutdown without
    /// waiting for process reaping. The kernel reclaims the assertion on
    /// process exit either way; this is a courtesy, not a leak fix.
    func teardown() {
        releaseAssertion()
    }

    /// Brings the held IOPMAssertion in line with `isEnabled` and persists the
    /// resolved value. If acquire fails, `isEnabled` is reverted to false so
    /// the toggle UI matches the actual sleep-prevention state.
    private func applyEnabled() {
        guard !suppressApply else { return }
        if isEnabled {
            if !acquireAssertion() {
                suppressApply = true
                isEnabled = false
                suppressApply = false
            }
        } else {
            releaseAssertion()
        }
        UserDefaults.standard.set(isEnabled, forKey: Self.preferencesKey)
    }

    private func acquireAssertion() -> Bool {
        guard !hasAssertion else { return true }
        var id: IOPMAssertionID = 0
        let reason = "Plow keeping Mac awake so the OpenClaw VM stays reachable" as CFString
        let result = IOPMAssertionCreateWithName(
            kIOPMAssertionTypePreventSystemSleep as CFString,
            IOPMAssertionLevel(kIOPMAssertionLevelOn),
            reason,
            &id
        )
        guard result == kIOReturnSuccess else {
            NSLog("[KeepMacAwake] IOPMAssertionCreateWithName failed: %d", result)
            return false
        }
        assertionID = id
        hasAssertion = true
        return true
    }

    private func releaseAssertion() {
        guard hasAssertion else { return }
        IOPMAssertionRelease(assertionID)
        hasAssertion = false
        assertionID = 0
    }
}

/// One-time cleanup of artifacts written by the pre-Lima (VZ) runtime. Existing
/// users upgrading from a VZ build carry 200MB–1GB of dead disk Lima never
/// touches. These paths are fully owned by the removed VMManager /
/// GatekeeperVMManager and have no live readers under the Lima runtime, so
/// unconditional removal is safe. Idempotent: absent paths are a no-op.
///
/// Targeted paths (all under `<appSupport>/`):
///   - `vm/rootfs.img` — writable copy of the OpenClaw VZ disk image.
///   - `gatekeeper-vm-state/` — VZ Gatekeeper instance directory (rootfs +
///     metadata), named by the removed GatekeeperVMManager.
///   - `gatekeeper-console.log` — VZ Gatekeeper serial-console log.
///
/// Downloaded gatekeeper tarballs in `runtime-cache/` are already reaped by
/// `RuntimeCacheCleaner.prune`, which drops any file not named in the current
/// manifest — so they don't need to be listed here.
enum LegacyVZArtifactCleanup {
    static func run() {
        run(appSupportRoot: AppPaths.appSupportRoot())
    }

    static func run(appSupportRoot: URL) {
        let fm = FileManager.default
        let targets: [URL] = [
            appSupportRoot.appendingPathComponent("vm/rootfs.img", isDirectory: false),
            appSupportRoot.appendingPathComponent("gatekeeper-vm-state", isDirectory: true),
            appSupportRoot.appendingPathComponent("gatekeeper-console.log", isDirectory: false),
        ]
        for url in targets {
            guard fm.fileExists(atPath: url.path) else { continue }
            let size = (try? fm.attributesOfItem(atPath: url.path)[.size] as? Int64) ?? -1
            do {
                try fm.removeItem(at: url)
                NSLog("[LegacyCleanup] removed stale VZ artifact path=%@ size=%lld", url.path, size)
            } catch {
                NSLog("[LegacyCleanup] failed to remove path=%@ error=%@", url.path, error.localizedDescription)
            }
        }
    }
}

// Every symbol is resolved at runtime so the app degrades gracefully
// if Apple removes these deprecated symbols in a future macOS release.

private struct _SharedFileListAPI {
    private let listCreate: ListCreateFn
    private let listCopySnapshot: ListCopySnapshotFn
    private let itemCopyResolvedURL: ItemCopyResolvedURLFn
    private let listInsertItemURL: ListInsertItemURLFn
    private let listItemRemove: ListItemRemoveFn
    private let favoritesID: UnsafeRawPointer
    private let itemLast: UnsafeRawPointer

    // C function signatures
    private typealias ListCreateFn = @convention(c) (
        CFAllocator?, CFString, CFTypeRef?
    ) -> UnsafeMutableRawPointer? // LSSharedFileListRef (Create rule)
    private typealias ListCopySnapshotFn = @convention(c) (
        UnsafeMutableRawPointer, UnsafeMutablePointer<UInt32>?
    ) -> Unmanaged<CFArray>?
    private typealias ItemCopyResolvedURLFn = @convention(c) (
        UnsafeMutableRawPointer, UInt32, UnsafeMutablePointer<Unmanaged<CFError>?>?
    ) -> Unmanaged<CFURL>?
    private typealias ListInsertItemURLFn = @convention(c) (
        UnsafeMutableRawPointer, UnsafeRawPointer?, CFString?, UnsafeRawPointer?, CFURL,
        CFDictionary?, CFArray?
    ) -> UnsafeMutableRawPointer?
    private typealias ListItemRemoveFn = @convention(c) (
        UnsafeMutableRawPointer, UnsafeMutableRawPointer
    ) -> OSStatus

    /// Returns nil if any required symbol is missing.
    init?() {

        guard let createSym = dlsym(handle, "LSSharedFileListCreate"),
              let snapshotSym = dlsym(handle, "LSSharedFileListCopySnapshot"),
              let resolveSym = dlsym(handle, "LSSharedFileListItemCopyResolvedURL"),
              let insertSym = dlsym(handle, "LSSharedFileListInsertItemURL"),
              let removeSym = dlsym(handle, "LSSharedFileListItemRemove"),
              let favSym = dlsym(handle, "kLSSharedFileListFavoriteItems"),
              let lastSym = dlsym(handle, "kLSSharedFileListItemLast")
        else {
            dlclose(handle)
            return nil
        }

        listCreate = unsafeBitCast(createSym, to: ListCreateFn.self)
        listCopySnapshot = unsafeBitCast(snapshotSym, to: ListCopySnapshotFn.self)
        itemCopyResolvedURL = unsafeBitCast(resolveSym, to: ItemCopyResolvedURLFn.self)
        listInsertItemURL = unsafeBitCast(insertSym, to: ListInsertItemURLFn.self)
        listItemRemove = unsafeBitCast(removeSym, to: ListItemRemoveFn.self)
        // These are pointers to global CF constants; dereference once to get the value.
        favoritesID = favSym.load(as: UnsafeRawPointer.self)
        itemLast = lastSym.load(as: UnsafeRawPointer.self)

        dlclose(handle)
    }

    func isInFavorites(url: CFURL) -> Bool {
        guard let (list, snapshot, _) = findItem(matching: url) else { return false }
        _ = snapshot
        Unmanaged<AnyObject>.fromOpaque(list).release()
        return true
    }

        guard let list = openFavoritesList() else { return }

        if let inserted = listInsertItemURL(list, itemLast, nil, nil, url, nil, nil) {
            Unmanaged<AnyObject>.fromOpaque(inserted).release()
        }
    }

    func removeFromFavorites(url: CFURL) {
        guard let (list, snapshot, item) = findItem(matching: url) else { return }
        defer { Unmanaged<AnyObject>.fromOpaque(list).release() }
        _ = listItemRemove(list, item)
        _ = snapshot // prevent release until after remove
    }

    private func openFavoritesList() -> UnsafeMutableRawPointer? {
        let cfID = Unmanaged<CFString>.fromOpaque(favoritesID).takeUnretainedValue()

    }

    /// The returned list is retained — caller must release.

            return nil

                if (resolved as URL).standardizedFileURL == targetURL {
                    return (list, snapshot, item)
                }
            }
        }
        Unmanaged<AnyObject>.fromOpaque(list).release()
        return nil
    }

# Phoenix + OpenClaw Security Architecture (Current State and Hardening Roadmap)

Last updated: 2026-03-03

## Purpose

This document describes:
1. The architecture Phoenix is running today.
2. What security guarantees exist right now.
3. What is not guaranteed yet.
4. A phased hardening roadmap.

This is a living architecture document for implementation and product/security decisions.

## System Overview (Current)

Core components:
1. `Phoenix.app` (macOS UI client).
2. `phoenixd` (local policy and orchestration daemon).
3. `openclaw-gateway` (tool runtime and agent loop, `exec.host = "gateway"` inside Alpine VM).
4. `phoenix-gateway-guard` OpenClaw plugin (`before_tool_call` / `after_tool_call` hooks).

Current runtime shape:
1. Gateway is bound to loopback (`127.0.0.1`) with token auth.
2. PhoenixD manages OpenClaw config reconciliation and process supervision.
3. PhoenixD resolves exec approval events over gateway operator websocket.
4. Tool policy decisions are requested from PhoenixD by plugin hook callbacks.

## Trust Boundaries (Current)

1. UI does not need direct gateway token by design.
2. Gateway token is stored in user home (`~/.phoenix/openclaw/secrets/gateway-token`) with mode `0600`.
3. Gateway/API access is local-only by bind setting, but same-user processes remain in the trust boundary.
4. OpenClaw runtime (gateway + node host) is currently trusted to honor plugin/approval contracts.

## Request Flow (Current)

Non-exec tool call:
1. OpenClaw gateway receives tool request.
2. Plugin `before_tool_call` calls PhoenixD `/internal/gateway/policy/tool-check`.
3. If denied or unavailable, plugin blocks (fail closed).
4. If allowed, OpenClaw gateway executes the tool.
5. Plugin `after_tool_call` posts result to PhoenixD audit endpoint.

Exec tool call:
1. Same `before_tool_call` policy gate applies.
2. Gateway raises `exec.approval.requested`.
3. PhoenixD operator client approves or denies.
4. If approved, execution happens via OpenClaw node host (`system.run`), not inside PhoenixD process.

## What Is Secure Today

1. Default local exposure is reduced (`loopback` + token auth).
2. Policy checks for tool calls are centralized in PhoenixD (plugin gate).
3. Exec has a second approval layer (`ask=always`, fallback deny behavior).
4. PhoenixD owns config reconciliation and can enforce baseline guardrails.
5. Operational audit trail exists for tool checks/results and process status.

## What Is Not Guaranteed Yet

1. Full runtime isolation from OpenClaw bugs is not guaranteed.
2. Gateway network egress is not fully forced through Phoenix today.
3. Same-user token theft is still possible on a compromised local account.
4. OpenClaw runtime remains a trusted enforcement participant in current model.
5. OS-level sandbox boundaries are not yet applied to gateway/node runtime paths.
6. `agentId` is not currently a cryptographically attested identity primitive and should not be used as a sole authorization key.
7. Plugin->Phoenix policy callbacks (`/internal/gateway/policy/*`) are local but currently do not have an additional dedicated auth secret/signature layer.

## Principal Risks

1. OpenClaw bypass bug risk:
   - A severe runtime bug could bypass plugin/approval paths.
2. Gateway direct egress risk:
   - Gateway may call model/provider/channels directly unless egress is constrained.
3. Same-user token risk:
   - Any process running as the same macOS user can potentially read the gateway token file.
4. Control-plane surface risk:
   - Any client with token and loopback access can attempt privileged gateway operations.
5. Identity confusion risk:
   - Policy decisions based only on payload-level `agentId` can be weaker than decisions keyed by stronger session/channel context.

## Identity Signals and Trust Strength (Current)

Signals currently observed in exec approval payloads include `agentId`, `sessionKey`, `nodeId`, and `systemRunBindingV1`.

Current trust guidance:
1. Treat `agentId` as a UX/debug attribute, not the sole security principal.
2. Treat `(nodeId, sessionKey)` as the primary policy key in current architecture.
3. Enforce consistency checks across duplicate fields (for example, `request.sessionKey` and `request.systemRunBindingV1.sessionKey`) and deny on mismatch.
4. Preserve fail-closed behavior for ambiguous or missing identity fields.

## Hardening Roadmap

### Phase 0: Baseline discipline (now)

1. Keep loopback bind and token auth mandatory.
2. Keep plugin required and fail-closed for policy check.
3. Keep exec ask/approval as deny-by-default fallback.
4. Keep OpenClaw version pinning and deterministic config reconciliation.
5. Prefer policy mode `ask` after approval UI is live.

### Phase 1: UX and approval authority hardening

1. Add PhoenixD <-> app websocket channel for real-time approvals.
2. Move default approval mode from auto-allow to ask.
3. Improve retry behavior around first exec/pairing race.
4. Add token rotation command and operational runbook.
5. Add explicit principal mapping and mismatch-deny logic keyed by `(nodeId, sessionKey)`.

### Phase 2: Packaging and deterministic runtime

1. Bundle pinned Node + OpenClaw runtime in signed installer payload.
2. Remove end-user npm/pnpm install dependencies.
3. Ensure daemon startup and bootstrap are deterministic on fresh machine.
4. Migrate startup ownership to launchd/SMAppService model.
5. Add plugin->Phoenix policy endpoint authentication (shared secret and/or signed request envelope).

### Phase 3: Execution isolation

1. Introduce Phoenix-managed sandboxed executor helper for exec path.
2. Route approved exec through sandbox helper instead of broad host runtime.
3. Enforce file/network/process limits at OS sandbox boundary.

### Phase 4: Egress and runtime containment

1. Enforce proxy-only provider/network path where gateway cannot bypass Phoenix proxy.
2. Add strong egress controls for gateway runtime.
3. Move toward full gateway/node isolation (containerized boundary or VM).
4. Optionally adopt custom Phoenix node runtime if protocol maintenance cost is acceptable.

## Strategic Options and Tradeoffs

Option A: Keep official OpenClaw node host + sandboxed exec helper
1. Lower maintenance cost.
2. Faster path to meaningful hardening.
3. Still trusts OpenClaw for non-exec behavior unless further containment is added.

Option B: Build custom Phoenix node runtime
1. Stronger control over exec enforcement.
2. Higher long-term protocol compatibility and maintenance burden.
3. Best used after installer and sandbox foundations are stable.

Option C: Full runtime containment (VM-level)
1. Strongest isolation model.
2. Highest complexity and operational cost.
3. Appropriate for high-assurance security target.

## Decision Guidance

Recommended near-term order:
1. Finish approval websocket/UI path.
2. Ship deterministic installer/runtime packaging.
3. Implement sandboxed exec helper.
4. Add proxy-only egress enforcement.
5. Reassess need for custom node runtime after measuring residual risk.

## Open Questions

1. What minimum security bar is required for first public installer release?
2. Is same-user token storage acceptable until Keychain migration, or should Keychain be mandatory before release?
3. Do we treat non-exec gateway egress lock-down as release-blocking or phase 2 hardening?
4. At what point does custom node runtime ROI exceed maintenance cost?

# 1Password Host-Side Broker Spike — Findings

## Purpose

The spike asked one question: can an openclaw agent in the Plow VM retrieve a user's 1Password credential by routing reads through a host-side `op` CLI broker that authenticates via the desktop app's biometric integration, with no service account? The plan ([`docs/superpowers/plans/2026-04-27-onepassword-broker-spike.md`](../superpowers/plans/2026-04-27-onepassword-broker-spike.md)) covered eight implementation tasks plus manual validation. ## Headline result

It works. The agent drove a full login at `practicetestautomation.com/practice-test-login/` using the bundled `PlowSpikeTest` Login item — listed candidates, matched URL, fetched after Touch ID, typed both fields, redirect to `/logged-in-successfully/` confirmed end-to-end success. Re-ran against the head chef's real Amazon credential. Same outcome. Architecture is sound.

## Plan's "Known fragilities" — verified

Two of three need no change for v1. **`whoami()` substring matching:** the exact `op` v2 stderr text wasn't captured, but the user-facing recovery from a signed-out state worked, so the not-signed-in branch was reached. **`_pick_field` purpose-based discrimination:** PlowSpikeTest and the Amazon item both returned populated `username`/`password` via `purpose: USERNAME`/`PASSWORD` — the plan's label-fallback hedge can stay deferred. **Plugin registration in `gateway.py`:** mirrored `plow-cli` correctly; gateway tests in `129c006d`, load-paths ratchet in `8a92a959`, rendered-Lima assertions in `0886f365` lock the wiring against silent regression.

## What the plan didn't anticipate

End-to-end exercise found two real bugs that two rounds of codex review missed.

**SKILL.md never reached the live workspace.** `plowd/gateway.py:828–846` carries a hardcoded allow-list of skill files to write; Task 8 placed the file under `templates/workspace/skills/` without registering it there. `35cebb82` added the entry; `ab5abd21` narrowed staging to fire only when `onepassword_plugin_dir is not None`. The same allow-list silently drops `plow-vip-contacts` and `plow-calendar` — broader bug class tracked in [#529](https://github.com/cncorp/plow/issues/529).

**Wrong execute() arity.** Tools were registered with `async execute(params)` but openclaw invokes `execute(toolCallId, params)`. The toolCallId string occupied the params slot, `params.item_id` was `undefined`, `JSON.stringify` dropped the key, FastAPI 422'd. Reference shape was sitting next door (`plow-cli/index.js:104`); Task 7 didn't follow it. `7851f964` matched the convention with a regression test; `a14ae7f6` wired previously-orphaned plugin JS test files into `just test-fast`. Lesson: any plan adding an openclaw plugin should call out the JS-runtime calling convention as an explicit review checkpoint, and any plugin's `index.test.js` should land in test-fast from day one.

## Privacy / log-audit

Audit of `plowd.log` (1819 lines, full session) showed zero credential leakage. Searches for the practice-site password, the practice URL, the practice username, the Amazon username, and generic `username=` / `password=` / `credential=` / common-email-domain patterns all returned zero hits in any broker-related context. The success path is silent. ## Architectural framing — paths forward

After validation, a deeper question reframed the productization conversation: should the agent ever see a credential index, and should every credential disclosure require explicit human approval? Today's spike answers "the agent sees the full sanitized index" and "first fetch prompts; subsequent fetches within the desktop app's ~10-minute auth cache are silent." Both are choices, not necessities.

Research into 1Password's official **Agentic Autofill** (Early Access since 2025-10-08; [developer docs](https://developer.1password.com/docs/agentic-autofill/), [blog post](https://1password.com/blog/closing-the-credential-risk-gap-for-browser-use-ai-agents)) shows a stark contrast. 1P's product builds three structural properties our spike does not: the agent never sees an index (no `list_logins` equivalent), the agent never sees plaintext (the 1P browser extension injects directly into the form DOM via a Noise-encrypted channel — the LLM tool-result never carries the secret), and every fetch is a separate human approval (no session cache). 1P explicitly calls out that "Using MCP exposes credentials directly to the underlying model or system, breaking our security principles" — our spike's shape is what they're rejecting. Their offering is closed today: the desktop app validates the pairing partner is `director.ai` and "rejects any untrusted party"; third-party agent runtimes can't plug in. We can replicate the *shape* using standard `op` primitives, but only with structural changes.

Four paths, in rough order:

**Path A — tighten the disclosure window to per-action approval.** Tactical hardening: shrink the `op` cache so any fetch outside an active login flow re-triggers biometric. **Investigated this branch; not implemented.** `op` v2.31.1 exposes no per-command re-auth mechanism — no `--reauth`, no `--no-cache` for the auth cache (`--cache=false` controls info caching only), no `OP_FORCE_PROMPT`-style env var. `OP_BIOMETRIC_UNLOCK_ENABLED` is a feature toggle, not a force-prompt. The only programmatic primitive is `op signout`, whose effect on desktop-integration mode is undocumented (auth state lives in the desktop app, not the CLI) and whose verification would disrupt the live session. The desktop app's "Authorization for command-line use" dropdown almost certainly has no sub-minute setting. Shipping a `signout`-before-fetch hack would risk surprising the user with re-prompts in their other terminals and still wouldn't deliver a deterministic window. Re-evaluate when 1P ships finer-grained auth control or when we move to Path C.

**Path B — move URL→item match server-side.** Broker takes a URL, returns one matching credential or null. `onepassword_list_logins` disappears from the agent tool surface. Eliminates **index disclosure** — arguably the larger of the two privacy concerns since the index reveals which sites the user has accounts on (metadata often as sensitive as the credentials themselves). Agent still sees plaintext for the one fetched credential. Estimated 1–2 weeks: URL-match logic in `onepassword.py`, retire the index tool, update SKILL.md to a single-step "give the broker a URL" pattern. Largest privacy win for the engineering investment.

**Path C — match the full 1P-shape: never put credentials in agent context.** Both URL match AND form-fill move out of the agent's tool path. Openclaw's browser-fill primitive (or a native bridge) requests a disclosure from a host-side service; user approves; credential is injected into browser DOM without passing through the LLM. Requires extending openclaw's browser-fill, building a native consent surface, and a trust contract between broker and browser instance. Weeks of work. The right end-state, but probably not the immediate next step.

**Path D — engage 1Password directly.** EAP language suggests they're soliciting integrators. If they open the Agentic Autofill protocol to third-party agent runtimes, Plow could become a Browserbase peer rather than reinventing the consent surface. Long lead time; lowest engineering tax if it works out. Worth pursuing in parallel as a follow-up — the answer materially affects whether B/C are necessary.

Path A is blocked at the `op` v2 layer; revisit when 1P exposes finer-grained auth control. Path B is the highest-value architectural move for the cost — index disclosure is the bigger of the two privacy gaps. Path C is the right end-state but probably not next. Path D is worth pursuing in parallel.

## What we'd build first if this graduates

In order: (1) multi-account picker + diagnostic "test 1Password integration" button — the two day-one rough edges; (2) **Path B (server-side URL match)** — biggest structural privacy win for the cost, and reduces the disclosure surface to one credential at a time; (3) TOTP support; (4) per-vault filter. Defer everything else, including Path A, until the underlying `op` primitives offer a clean mechanism or until Path C is on the roadmap.

## Open questions

- **Cache-window timeout.** Mid-session expiry behavior unverified — we didn't wait the 10 min out.
- **Same-URL multi-item.** n=2 isn't n=many; real disambiguation frequency unknown.
- **`op` CLI redistribution licensing.** Gates bundling.
- **Plugin tools without SKILL.md.** Whether plugin-registered tools surface to the LLM schema in the absence of a corresponding SKILL.md was not isolated. Worth confirming before relying on tool registration as the sole agent-discovery mechanism.
- **`op signout` semantics under desktop integration.** Whether it invalidates the desktop app's biometric auth cache or only clears CLI-side state. Resolves whether Path A is salvageable as a hack.

## Known Review Mistakes (avoid repeating these)\n# Comment Review Mistakes

Patterns where the automated reviewer has over-called, mis-calibrated, or otherwise got it wrong.

Each entry is a **pattern**, not a case study: no specific PR numbers, no specific file paths, no team-specific details. The goal is a durable principle future reviews will apply.

1. Don't flag missing tests as blocking when 1–2 focused behavior tests or smoke checks cover the user-visible risk. A change needs proof the behavior holds, not layer-by-layer coverage.
2. Don't require new tests for PRs that only change documentation, manifests, workflow YAML, or artifact pins. Note gaps as low/nit at most.
3. APPROVE when there are no findings medium or higher. Low, and nit findings are noted in the review but do not prevent approval. Only `blocking` `high` and `medium` -tagged findings warrant `VERDICT: COMMENT`.
4. When approving with nit/low findings, prefer `VERDICT: APPROVE — pending: <nit1>, <nit2>` so the approval body communicates what minor things would be nice to clean up. e.g. "Approving — pending: tighten empty-string guard in pushSlackAccount, add one argv-ordering test"

6. Don't propose replacing `assert X` (or other fail-fast guards) with explicit raise + logging context. The user's `Fail-Fast` standard prefers the assertion crashing loudly on internal-caller bugs; logging+raise is bloat unless the code sits at a trust boundary.
7. Don't propose `isinstance` / type-validation checks for internal callers. Validation belongs at trust boundaries (user input, external APIs); internal calls trust their callers — adding type guards calcifies the call shape and contradicts `Concise Code`.
8. Don't propose state-reset / fallback writes ("set X = default in case prior init left it dirty") unless the polluting scenario is observed in production, not theoretical. 9. Don't propose wrapper dataclasses, snapshot views, or new DI seams when a 1–2 line direct fix solves the same problem. Snapshots and freeze-views are remedies for *repeated* problems, not first instances; "Three similar lines > premature abstraction" applies.
10. Don't propose streaming / incremental rewrites of small in-memory operations on theoretical perf or OOM grounds. Cite a measured failure or skip — the streaming version typically adds 15+ LOC of state machine, and the in-memory version costs nothing on the inputs that actually arrive.
11. Don't propose adding companion tests, CI guards, or regression coverage for scenarios that haven't been shown to actually occur ("no test for prod re-enable", "no test that this can't happen").     76	- Grep the repo for similar function names and near-duplicate strings.
    77	- If a similar implementation exists, unify rather than adding alongside.
    78	- Duplicate logic with an existing repo site: *blocking: dry* — cite the existing site by path.
    79	- Intra-PR duplication (N near-identical blocks in the same diff) is a missing abstraction: *medium: dry*.
    80	
    81	**Review question:** *Is this DRY with the rest of the codebase? Are we re-using existing functions where we should?*
    82	
    83	## Regression Risk
    84	
    85	New code must not quietly change existing business logic. Any change to a function's contract, return shape, side effects, or error behavior needs an explicit callout.
    86	
    87	- Changed semantics of an existing function without updating call sites: *blocking: regression*.
    88	- New conditional branch that alters legacy-path behavior: *medium: regression* unless proven safe.
    89	- A bug fix without a regression test that exercises the old-bug path is *blocking: tests*.
    90	
    91	**Review question:** *Does this introduce regressions? How does it change existing business logic, and is that change intentional and covered?*
    92	
    93	## Reframe the Spec
    94	
    95	Any single new block over ~20 LOC is a smell. Pause and ask whether a spec or UX tweak would delete most of that code. Don't silently accept contorted code to match a literal ask — name the simpler alternative so the author can choose.
    96	
    97	- If satisfying a spec detail is adding notable complexity: *low: spec-reframe* with the alternative described in one sentence.
    98	- If the complexity is structural (not just a spec ask): escalate to *medium*.
    99	
   100	**Review question:** *Would a small change to the spec or UX accomplish the spirit of the request while significantly reducing complexity?*
   101	
   102	## Name the Shape
   103	
   104	Before writing code, name the *class* of problem: parsing, validation, dispatch, retry, auth, serialization, formatting, batching, state, idempotency, audit logging, feature-flag gating. Each is a recurring shape with a preferred home in the codebase. Pick the shape first; the lines come second.
   105	
   106	Two failure modes:
   107	
   108	- **Wrong shape.** Regex when the upstream could emit structured data. Hand-rolled validation when pydantic/zod is already in use. `if kind == "A": ... elif kind == "B": ...` instead of a dispatch dict. Bool-soup state instead of an enum. The code works on today's input and silently breaks on the next variant. Regex is the canonical example — treat regex on string-typed input as a smell that *structure got discarded upstream*. The fix is usually to make upstream emit data, not to grow the regex.
   109	- **Five shapes for five similar problems.** Each new feature lands with its own auth check, its own retry loop, its own date format, its own error shape. Each PR looks fine in review; the policy now lives in 17 places. If you are the second instance of a missing shape, *that* is the moment to introduce one — not the fifth.
   110	
   111	Before writing: grep for how the codebase already handles this class. If a shape exists, conform. If none exists and you are not the first instance, you have just discovered tech debt — flag it.
   112	
   113	**Review question:** *Did the author name the problem class and conform to (or establish) the codebase's shape for it, or did they solve only the literal symptom?*
   114	
   115	## Generalize the Fix (Narrow-Fix)
   116	
   117	A fix tied to one reported input (regex, string equality, hard-coded special case) is a narrow fix. Ask: what is the next variant of this input, and does this code cover it?
   118	
   119	- Narrow fixes that won't survive the obvious next bug are *medium: narrow-fix*.
   120	- If the root cause is systemic (parsing, state machine, schema), fix the class, not the instance.
