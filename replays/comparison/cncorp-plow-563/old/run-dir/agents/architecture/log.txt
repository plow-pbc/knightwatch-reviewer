## PR Description (author's own explanation)

## Summary

- Adds a "Keep Mac Awake" toggle (Phoenix menu-bar app) backed by `IOPMAssertionCreateWithName` with `kIOPMAssertionTypePreventSystemSleep`, so the agent stays reachable while the Mac is unattended. State persists via `UserDefaults`; the assertion is held for the lifetime of the toggle being on, and released on `applicationWillTerminate`.
- Toggle surfaces in two places: the installer's Settings card (its own "System" section, separate from Connectors) and the connected status popover.
- On acquire failure the toggle reverts to OFF so the UI can't lie about whether sleep is actually being prevented; failures are logged via `NSLog`.
- Installer window now auto-sizes its height to settings content + `DownloadBarView` overlay, with a screen-height clamp and `ScrollView` fallback for unusually-tall content.

## Implementation notes

- `KeepMacAwake` is `@MainActor`, `@Observable`, with `isEnabled` as a stored property so all observers re-render on toggle.
- Two SwiftUI `PreferenceKey`s drive window sizing: `InstallerContentHeightKey` measures the settings VStack, `InstallerOverlayHeightKey` measures the download bar.
- Iterated through two rounds of cross-model code review (claude implementer + codex adversarial reviewer).

# Plow

Plow is an agent platform that gives OpenClaw agents narrow, permission-gated access to cloud services (Gmail, Slack) and on-device capabilities through a local approval layer.

The system has three tiers:

- **Plow API** (`api/`) — cloud service that owns connector OAuth, opportunity matching, marketplace skills, and policy enforcement
- **Plow app** (`app/`) — macOS app (Swift) + plowd daemon (Python) that manages on-device approval, gateway configuration, and VM lifecycle
- **Plow CLI** (`cli/`) — narrow tool surface inside the OpenClaw container, talks directly to the Plow API

# Phoenix + OpenClaw Security Architecture (Current State and Hardening Roadmap)

Last updated: 2026-03-03

## Purpose

This document describes:
1. The architecture Phoenix is running today.
2. What security guarantees exist right now.
3. What is not guaranteed yet.
4. A phased hardening roadmap.

This is a living architecture document for implementation and product/security decisions.

## System Overview (Current)

Core components:
1. `Phoenix.app` (macOS UI client).
2. `phoenixd` (local policy and orchestration daemon).
3. `openclaw-gateway` (tool runtime and agent loop, `exec.host = "gateway"` inside Alpine VM).
4. `phoenix-gateway-guard` OpenClaw plugin (`before_tool_call` / `after_tool_call` hooks).

Current runtime shape:
1. Gateway is bound to loopback (`127.0.0.1`) with token auth.
2. PhoenixD manages OpenClaw config reconciliation and process supervision.
3. PhoenixD resolves exec approval events over gateway operator websocket.
4. Tool policy decisions are requested from PhoenixD by plugin hook callbacks.

## Trust Boundaries (Current)

1. UI does not need direct gateway token by design.
2. Gateway token is stored in user home (`~/.phoenix/openclaw/secrets/gateway-token`) with mode `0600`.
3. Gateway/API access is local-only by bind setting, but same-user processes remain in the trust boundary.
4. OpenClaw runtime (gateway + node host) is currently trusted to honor plugin/approval contracts.

## Request Flow (Current)

Non-exec tool call:
1. OpenClaw gateway receives tool request.
2. Plugin `before_tool_call` calls PhoenixD `/internal/gateway/policy/tool-check`.
3. If denied or unavailable, plugin blocks (fail closed).
4. If allowed, OpenClaw gateway executes the tool.
5. Plugin `after_tool_call` posts result to PhoenixD audit endpoint.

Exec tool call:
1. Same `before_tool_call` policy gate applies.
2. Gateway raises `exec.approval.requested`.
3. PhoenixD operator client approves or denies.
4. If approved, execution happens via OpenClaw node host (`system.run`), not inside PhoenixD process.

## What Is Secure Today

1. Default local exposure is reduced (`loopback` + token auth).
2. Policy checks for tool calls are centralized in PhoenixD (plugin gate).
3. Exec has a second approval layer (`ask=always`, fallback deny behavior).
4. PhoenixD owns config reconciliation and can enforce baseline guardrails.
5. Operational audit trail exists for tool checks/results and process status.

## What Is Not Guaranteed Yet

1. Full runtime isolation from OpenClaw bugs is not guaranteed.
2. Gateway network egress is not fully forced through Phoenix today.
3. Same-user token theft is still possible on a compromised local account.
4. OpenClaw runtime remains a trusted enforcement participant in current model.
5. OS-level sandbox boundaries are not yet applied to gateway/node runtime paths.
6. `agentId` is not currently a cryptographically attested identity primitive and should not be used as a sole authorization key.
7. Plugin->Phoenix policy callbacks (`/internal/gateway/policy/*`) are local but currently do not have an additional dedicated auth secret/signature layer.

## Principal Risks

1. OpenClaw bypass bug risk:
   - A severe runtime bug could bypass plugin/approval paths.
2. Gateway direct egress risk:
   - Gateway may call model/provider/channels directly unless egress is constrained.
3. Same-user token risk:
   - Any process running as the same macOS user can potentially read the gateway token file.
4. Control-plane surface risk:
   - Any client with token and loopback access can attempt privileged gateway operations.
5. Identity confusion risk:
   - Policy decisions based only on payload-level `agentId` can be weaker than decisions keyed by stronger session/channel context.

## Identity Signals and Trust Strength (Current)

Signals currently observed in exec approval payloads include `agentId`, `sessionKey`, `nodeId`, and `systemRunBindingV1`.

Current trust guidance:
1. Treat `agentId` as a UX/debug attribute, not the sole security principal.
2. Treat `(nodeId, sessionKey)` as the primary policy key in current architecture.
3. Enforce consistency checks across duplicate fields (for example, `request.sessionKey` and `request.systemRunBindingV1.sessionKey`) and deny on mismatch.
4. Preserve fail-closed behavior for ambiguous or missing identity fields.

## Hardening Roadmap

### Phase 0: Baseline discipline (now)

1. Keep loopback bind and token auth mandatory.
2. Keep plugin required and fail-closed for policy check.
3. Keep exec ask/approval as deny-by-default fallback.
4. Keep OpenClaw version pinning and deterministic config reconciliation.
5. Prefer policy mode `ask` after approval UI is live.

### Phase 1: UX and approval authority hardening

1. Add PhoenixD <-> app websocket channel for real-time approvals.
2. Move default approval mode from auto-allow to ask.
3. Improve retry behavior around first exec/pairing race.
4. Add token rotation command and operational runbook.
5. Add explicit principal mapping and mismatch-deny logic keyed by `(nodeId, sessionKey)`.

### Phase 2: Packaging and deterministic runtime

1. Bundle pinned Node + OpenClaw runtime in signed installer payload.
2. Remove end-user npm/pnpm install dependencies.
3. Ensure daemon startup and bootstrap are deterministic on fresh machine.
4. Migrate startup ownership to launchd/SMAppService model.
5. Add plugin->Phoenix policy endpoint authentication (shared secret and/or signed request envelope).

### Phase 3: Execution isolation

1. Introduce Phoenix-managed sandboxed executor helper for exec path.
2. Route approved exec through sandbox helper instead of broad host runtime.
3. Enforce file/network/process limits at OS sandbox boundary.

### Phase 4: Egress and runtime containment

1. Enforce proxy-only provider/network path where gateway cannot bypass Phoenix proxy.
2. Add strong egress controls for gateway runtime.
3. Move toward full gateway/node isolation (containerized boundary or VM).
4. Optionally adopt custom Phoenix node runtime if protocol maintenance cost is acceptable.

## Strategic Options and Tradeoffs

Option A: Keep official OpenClaw node host + sandboxed exec helper
1. Lower maintenance cost.
2. Faster path to meaningful hardening.
3. Still trusts OpenClaw for non-exec behavior unless further containment is added.

Option B: Build custom Phoenix node runtime
1. Stronger control over exec enforcement.
2. Higher long-term protocol compatibility and maintenance burden.
3. Best used after installer and sandbox foundations are stable.

Option C: Full runtime containment (VM-level)
1. Strongest isolation model.
2. Highest complexity and operational cost.
3. Appropriate for high-assurance security target.

## Decision Guidance

Recommended near-term order:
1. Finish approval websocket/UI path.
2. Ship deterministic installer/runtime packaging.
3. Implement sandboxed exec helper.
4. Add proxy-only egress enforcement.
5. Reassess need for custom node runtime after measuring residual risk.

## Open Questions

1. What minimum security bar is required for first public installer release?
2. Is same-user token storage acceptable until Keychain migration, or should Keychain be mandatory before release?
3. Do we treat non-exec gateway egress lock-down as release-blocking or phase 2 hardening?
4. At what point does custom node runtime ROI exceed maintenance cost?

# File Taxonomy: Upstream vs Wrapper vs User State vs Team-Tracked

*Last updated: 2026-04-29*

Every file in the Plow universe belongs to exactly one of four categories.
Mixing them is the source of nearly every "broken on upgrade" or "shipped
user data" incident we hit.

The pre-commit guard at `scripts/check_protected_paths.py` enforces the rules
below by blocking edits to known protected paths; overrides are possible but
intentional.

## The four categories

| # | Category | Source of truth | On user's disk at runtime | Who modifies | Modify directly? |
|---|---|---|---|---|---|
| **(a)** | **Upstream OpenClaw runtime** — Gateway, ACP runner, cron engine, heartbeat loop, plugin SDK, Control UI | `openclaw/upstream` repo (NOT in this tree); pulled at VM build time via `npm install -g openclaw@<version>` (pinned in `app/vm/manifest.toml`) | Inside the Lima VM's OCI image at `/usr/local/lib/node_modules/openclaw/`. Never on the host FS. | Upstream maintainers | **No.** Any host-side edit is overwritten on the next VM image pull. Fix behavior in (b), or open a PR upstream and bump the pin. |
| **(b)** | **Plow wrapper code** that we author and ship | This repo: `app/Phoenix/` (Swift app), `app/plowd/plowd/` (Python daemon), `app/openclaw/plugins/{guard,plow-cli}/`, `app/openclaw/channels/plow-imessage/`, `app/openclaw/templates/workspace/`, `app/openclaw/marketplace/`, `app/vm/`, `api/plow/`, `cli/` | Packaged into `Plow.app/Contents/Resources/runtime/…` at build time | Plow engineers (us) via PR | **Yes** — this is the only layer we own. All fixes, features, new tools, guardrail changes belong here. |
| **(c)** | **Per-install user state** — per-machine config, history, secrets, workspace | Written at runtime by Plow or by the user; *never* committed to this repo | `~/Library/Application Support/co.plow.app/openclaw/…` (prod), `…/co.plow.app.dev.wt1/…` (dev variants), `~/.openclaw/` (standalone dev installs), `~/Plow/` (user's shared workspace folder), `~/.phoenix/` (PhoenixD state) | The running Plow app, or the user | **Rarely.** Only: (i) live-debugging *this* machine with explicit consent, (ii) a one-time first-launch migration written as (b) code, (iii) (b) code seeding defaults into fresh (c) state (e.g. `plowd/gateway.py::_apply_locked_guardrails`). Never check (c) data into the repo; never rely on a manual (c) edit to persist a behavior change. |
| **(d)** | **Team-tracked skill bundles** — opt-in skill bundles the team wants under git for review and history but explicitly NOT default-shipped. Currently `team-skills/`. Distinct from `app/openclaw/marketplace/` (which is (b) and ships with Plow.app). | This repo: `team-skills/` | Nowhere by default. Users who want a bundle copy or reference it into their own `/workspace/skills/` manually. | Plow engineers (us) via PR | **Yes** — same review process as (b), but no build/install pipeline picks (d) up automatically. The `skill-install-state` guard rule blocks (c) install metadata (`.clawhub/`, `_meta.json`) from leaking into `(d)` when copying a live skill. |

## VM recreate safety

The Lima VM can be destroyed and recreated without losing any user data.
All (c) state is on the host and mounted into the VM via virtiofs — confirmed
in `app/vm/lima.yaml`:

| Mount point | Host location placeholder | Writable | Category |
|---|---|---|---|
| `/workspace` | `__PLOW_WORKSPACE_DIR__` | yes | (c) user workspace |
| `/config` | `__PLOW_CONFIG_DIR__` | yes | (c) per-install config |
| `/mounts` | `__PLOW_MOUNTS_DIR__` | yes | (c) user-attached mounts |
| `/vm-logs` | `__PLOW_VM_LOGS_DIR__` | yes | (c) runtime logs |
| `/plugin`, `/plow-cli-plugin`, `/plow-imessage`, `/cli-packages`, `/images` | various | **no** | (b) shipped code, read-only |

Nothing user-valuable lives only inside the VM filesystem. The read-only
mounts are all (b) code that Phoenix re-provides from `Plow.app` on the next
launch. A `limactl delete` + re-provision is therefore safe at any time.

## Two rules that follow from the taxonomy

### Rule 1 — Never ship (c) data

Anything that ends up in `Plow.app/Contents/Resources/` must be (b) content.
Two paths cross the boundary and need the most vigilance:

1. **`app/openclaw/templates/workspace/`** — files here are copied into each
   user's fresh (c) workspace on first run. They must remain generic
   skeletons. Personalized variants (`HEARTBEAT.md`, populated `USER.md`)
   belong in (c), not (b).

2. **`app/openclaw/marketplace/`** — shipped cron recipes / skill bundles
   users can instantiate. Same rule: generic, no personal data.

### Rule 2 — Never modify (a)

There is no (a) source in this repo. The surface area for accidentally
"patching upstream" is at runtime on a dev's machine:

- `~/Library/Application Support/co.plow.app/openclaw/gateway/openclaw.json` —
  your personal generated config. Host-side edits are overwritten next
  launch by `plowd/gateway.py::_apply_locked_guardrails`. To change defaults,
  edit the generator in (b).
- `~/.openclaw/` — same pattern for the standalone dev install.
- `/usr/local/lib/node_modules/openclaw/` inside the VM — replaced on every
  VM image pull.

If you keep needing to make the same manual edit in (c), that's a missing
(b) feature. File it, don't repeat it.

## The protected-paths guard

`scripts/check_protected_paths.py` is wired into `.pre-commit-config.yaml` as
the `check-protected-paths` hook. It inspects each staged file against a
short list of rules; each rule has a stable `id` that can be used to
override the check for a single commit.

### Rule types

Two mechanisms work in parallel:

**Path-based rules** (`RULES` in `scripts/check_protected_paths.py`) block a
file from being staged at all when its path matches a glob. Override per
commit with `PLOW_GUARD_ALLOW=<rule-id>`.

**Skeleton-drift check** (`scripts/protected-skeleton-hashes.txt`) pins a
SHA256 for each shipped-skeleton file. Any modification that doesn't match
the pinned hash fails the check. There is **no** `PLOW_GUARD_ALLOW` for
drift on purpose: updating the pinned hash in the same commit *is* the
override, and it's visible to reviewers in the diff.

### Current rules

| `id` | Kind | Patterns / pin | Category | Override |
|---|---|---|---|---|
| `user-state` | path | `**/cron/jobs.json`, `**/cron/jobs.json.*`, `**/cron/runs/*.jsonl`, `**/sessions/store.json`, `**/agents/*/sessions/*.jsonl`, `**/gateway-token`, `**/node-device-id`, `**/plow-api-token`, `**/plow-local-token` | (c) per-machine runtime state | `PLOW_GUARD_ALLOW=user-state` |
| `personal-workspace-file` | path | `app/openclaw/templates/workspace/HEARTBEAT.md` | (b)→(c) personalized template | `PLOW_GUARD_ALLOW=personal-workspace-file` |
| `openclaw-upstream-pin` | path | `app/vm/manifest.toml` | (a) upstream version pin | `PLOW_GUARD_ALLOW=openclaw-upstream-pin` |
| `skill-install-state` | path | `**/.clawhub/**`, `team-skills/**/_meta.json` | (c) per-install skill metadata leaking into the repo | `PLOW_GUARD_ALLOW=skill-install-state` |
| `skeleton-drift` | hash | `app/openclaw/templates/workspace/USER.md`, `…/IDENTITY.md` | (b) canonical shipped skeleton | Update pin in `scripts/protected-skeleton-hashes.txt` in the same commit |

### Overriding a path rule for a single commit

When an edit is intentional — e.g. bumping the upstream pin after a
release, or force-adding `HEARTBEAT.md` for some legit reason:

```bash
PLOW_GUARD_ALLOW=openclaw-upstream-pin git commit -m "vm: bump openclaw to 2026.5.1"
PLOW_GUARD_ALLOW=personal-workspace-file,openclaw-upstream-pin git commit ...
```

Overrides are per-invocation; there is no persistent allow-list file on
purpose — the override should appear explicitly in the shell history and be
visible to reviewers via the commit's scope.

### Updating a pinned skeleton

You intentionally changed `USER.md` or `IDENTITY.md` in a way every Plow
user should see. To ship it:

```bash
# 1. Make your edit to the skeleton file.
# 2. Get the new hash.
shasum -a 256 app/openclaw/templates/workspace/USER.md
# 3. Update the matching line in scripts/protected-skeleton-hashes.txt
# 4. Stage both files and commit.
git add app/openclaw/templates/workspace/USER.md scripts/protected-skeleton-hashes.txt
git commit -m "templates(workspace): add <field> to USER.md skeleton"
```

The guard will pass because the recomputed SHA matches the new pin. The
diff on `scripts/protected-skeleton-hashes.txt` makes the intent explicit
to reviewers and forces a moment of reflection per this doc — that's the
entire point.

### Adding a new rule

**New path rule:** edit the `RULES` list in
`scripts/check_protected_paths.py` and add a row to the table above.
Fields: `id` (short kebab-case token), `category` (string tag), `patterns`
(glob list), `why` (failure message). Glob dialect: `**` (any number of
path components), `*` (non-slash chars), `?` (one non-slash char);
everything else literal.

**New skeleton pin:** add a line to
`scripts/protected-skeleton-hashes.txt` with the file's current SHA256 and
its repo-relative path, then add it to the table above.

# OpenClaw + Phoenix Architecture

*Last updated: 2026-03-04*

> **Historical.** This document describes the pre-Lima architecture: a custom Alpine
> rootfs booted via `Virtualization.framework` with vsock/socat bridges. On branch
> `vm/lima-migration`, OpenClaw now runs as an OCI container inside a Lima-managed
> Debian guest (`app/vm/lima.yaml`, `app/Phoenix/LimaVMManager.swift`), and the
> rootfs/initramfs/vsock machinery below is gone. Gatekeeper (Camoufox) still uses
> `Virtualization.framework` on this branch. See PR #310 for the Lima migration
> (openclaw + Debian guest) and the camoufox-on-lima follow-up.

## Ownership Map

Every component in this system is owned by exactly one of two codebases:

| Owner | Codebase | Language | What It Owns |
|-------|----------|----------|-------------|
| **OpenClaw** (upstream) | `openclaw/upstream` | Node.js / TypeScript | Gateway, Plugin SDK, Channels, Heartbeat, Cron, ACP, Sessions, Control UI, Pi agent runner |
| **Phoenix** (ours) | `app/`, `app/plowd/`, `app/vm/` | Python + Swift + JS plugins | PhoenixD daemon, macOS app, VM image, phoenix-gateway-guard plugin, plow-imessage channel plugin, config generation |

**Phoenix owns zero lines of the Gateway.** We treat OpenClaw as an opaque
runtime binary. Our integration surface is:

1. **VM management** — we run `openclaw gateway run` inside an Alpine Linux VM (Virtualization.framework)
2. **WebSocket operator protocol** — we connect as an operator to observe and approve
3. **Plugin hooks** — our JS plugin runs inside their Gateway process
4. **Channel plugin** — our plow-imessage channel plugin runs inside their Gateway process
5. **Config generation** — we write `openclaw.json` with locked guardrails

**Note:** We use `exec.host = "gateway"` — the Gateway executes commands directly
(no separate Node Host process). The VM itself provides sandboxing.

## VM Sandbox Architecture

### Why a VM?

- **Isolation**: The Gateway executes arbitrary shell commands on behalf of agents
  (`exec.host = "gateway"`). Running inside a VM limits blast radius — a compromised
  agent cannot access the host filesystem, keychain, or other processes.
- **Reproducibility**: The Alpine rootfs is built deterministically via Docker. Every
  install gets the same sandbox environment.
- **Transparency**: vsock + socat bridges make the VM boundary invisible to both
  PhoenixD and OpenClaw. Gateway still binds `127.0.0.1:18789`, PhoenixD still connects
  to that port — the bridges handle the translation.

### Boot Chain

```
VZLinuxBootLoader (direct kernel boot, no UEFI/bootloader)
  vmlinuz (Alpine linux-virt kernel)
  initramfs.img (custom, static busybox)
    → load kernel modules: virtio, ext4, vsock, virtiofs, virtio_net
    → mount /dev/vda (ext4 rootfs)
    → switch_root

  /sbin/init (PID 1, shell script)
    → mount /proc, /sys, /dev
    → ip link set lo up
    → bring up NAT interface (eth0) via DHCP
    → set deterministic DNS (8.8.8.8, 1.1.1.1)
    → mount virtiofs shares
    → bind-mount /vm-logs → /tmp/openclaw (shared log access)
    → start socat bridges (A + B)
    → signal host readiness via vsock :18787
    → wait for PhoenixD to seed /config/gateway/openclaw.json + gateway token
    → start openclaw-start.sh (gateway)
    → wait for gateway readiness on :18789 (up to 30s)
    → wait (PID 1 keeps running)
```

### Bridge Architecture

Four bridges connect host and guest. Guest-side bridges are socat (in init script).
Host-side bridges are Swift (in VMManager.swift) using NWListener and VZVirtioSocketDevice.

```
HOST (macOS)                                 VM (Alpine Linux)

Bridge C (Swift, VMManager)                  Bridge A (socat, init)
  vsock :18788 listener ◄────vsock────►  TCP-LISTEN:18788 → VSOCK-CONNECT:2:18788
  → relay to TCP 127.0.0.1:18788             (plugin HTTP → PhoenixD)

Bridge D (Swift, VMManager)                  Bridge B (socat, init)
  NWListener TCP :18789 ────vsock────►  VSOCK-LISTEN:18789 → TCP:127.0.0.1:18789
  → relay to vsock :18789                    (PhoenixD → Gateway)
```

**Traffic flows:**
1. PhoenixD (host :18788) ← Bridge C ← vsock ← Bridge A ← plugin HTTP (guest)
2. PhoenixD (host) → TCP :18789 → Bridge D → vsock → Bridge B → Gateway :18789 (guest)

Port 18787 is used only for the one-shot readiness signal (guest → host).

### VM Networking

The VM has outbound internet via `VZNATNetworkDeviceAttachment` (Apple's built-in NAT).
This is needed for channel plugins (e.g., plow-imessage WebSocket to `api.plow.co`).

**Requirements for outbound to work:**
- `virtio_net` kernel module loaded in initramfs (plus `failover`, `net_failover`)
- NAT interface brought up via `ip link set eth0 up` + `udhcpc` in init
- Deterministic DNS written to `/etc/resolv.conf` (DHCP DNS is unreliable)
- `ca-certificates` installed in the rootfs (for HTTPS)

### virtiofs Mounts

| Tag | Guest Path | Mode | Purpose |
|-----|-----------|------|---------|
| workspace | /workspace | rw | Agent working directory |
| config | /config | rw | Gateway config, secrets, sessions, agents |
| mounts | /mounts | rw | Additional host paths |
| plugin | /plugin | ro | phoenix-gateway-guard plugin (from app bundle) |
| channel | /plow-imessage | ro | plow-imessage channel plugin (from app bundle) |
| vm-logs | /vm-logs | rw | Shared log directory (bind-mounted to /tmp/openclaw) |
| desktop | /host/Desktop | rw | Host user's Desktop (when mounted) |

Plugin and channel dirs are compiled JS from the app bundle, mounted read-only.
Config must be rw — the Gateway writes `openclaw.json`, sessions, and secrets.
Host-share paths are strict: `/host/<Name>` exists only when the corresponding
share is actually mounted. The guest does not keep placeholder directories for
absent shares. Updating the share set requires a VM restart.
Phoenix JSON-encodes the active `[HostShare]` list into the
`PLOWD_OPENCLAW_ACTIVE_HOST_SHARES` environment variable before spawning
plowd; plowd reads that env var as its sole source of truth for which
host shares are live this session.

### rootfs Management

The rootfs image is copied from the signed app bundle to `~/.phoenix/vm/rootfs.img`
(writable location required by Virtualization.framework). VMManager auto-detects
bundle changes by comparing file sizes — if the bundle rootfs differs from the local
copy, it replaces the local copy on next boot. No manual deletion needed.

### VM Image Build

3-stage Dockerfile in `app/vm/build-vm-image`:

1. **builder** — Alpine + Node.js + `npm install -g openclaw@<version>`
2. **runtime** — Clean Alpine with `linux-virt` kernel, socat, Node.js, ca-certificates.
   Copies node_modules from builder. Copies init scripts.
3. **initramfs-builder** — Builds custom initramfs from runtime's kernel modules
   (static busybox, decompressed `.ko` files, custom init script)

Outputs: `app/vm/output/{vmlinuz, initramfs.img, rootfs.img}`

**Why custom initramfs?** Alpine's stock `initramfs-virt` uses OpenRC — incompatible
with Virtualization.framework's direct kernel boot. Our initramfs uses static busybox
and loads only the modules we need.

## Channel Plugin: plow-imessage

The `plow-imessage` channel plugin connects the OpenClaw agent to iMessage via the

```
iMessage user
    ↓ (Apple iMessage)
```

**Key files:**

     6	/// the system stays in Dark Wake and the OpenClaw VM keeps serving through
     7	/// display sleep. On battery the kernel effectively downgrades the assertion to
     8	/// idle-sleep-prevention, so a single assertion works in both power states;

    10	/// Clamshell sleep on a MacBook without an external display is still enforced
    11	/// by the kernel and isn't reachable from a public API.

    19	    /// Set true while we're rewriting `isEnabled` from inside `applyEnabled` to
    20	    /// revert a failed acquire — the recursive `didSet` reads this and bails.
    21	    private var suppressApply = false
    22

    24	        didSet { applyEnabled() }

    29	    }

    31	    /// Releases the assertion deterministically before the app exits, so the
    32	    /// system can coalesce any pending sleep state at logout/shutdown without
    33	    /// waiting for process reaping. The kernel reclaims the assertion on
    34	    /// process exit either way; this is a courtesy, not a leak fix.
    35	    func teardown() {
    36	        releaseAssertion()
    37	    }
    38	
    39	    /// Brings the held IOPMAssertion in line with `isEnabled` and persists the
    40	    /// resolved value. If acquire fails, `isEnabled` is reverted to false so
    41	    /// the toggle UI matches the actual sleep-prevention state.
    42	    private func applyEnabled() {
    43	        guard !suppressApply else { return }
    44	        if isEnabled {
    45	            if !acquireAssertion() {
    46	                suppressApply = true
    47	                isEnabled = false
    48	                suppressApply = false
    49	            }
    50	        } else {
    51	            releaseAssertion()
    52	        }
    53	        UserDefaults.standard.set(isEnabled, forKey: Self.preferencesKey)
    54	    }

## Team Context

Small team (~3 engineers), ~10 active users, moving fast. Design bias:

- **Elegance > defensive sprawl.** Code that reads cleanly and scales beats code that handles every hypothetical failure mode.
- **Concise code that fails loudly > verbose, defensive code with brittle special cases.** We prefer short code that captures the *spirit* of the spec or desired UX and crashes when its assumptions break, over long code that bakes in handling for every edge case the author imagined. Each special case calcifies a decision and invites the next one. When the spec seems to require contortions, the right reflex is to question the spec (see "Reframe the Spec" below), not to add branches.
- **Engineer time > compute time.** When picking between approaches, prefer the one that scales in *engineers' time* — fewer special cases, fewer conditional branches, fewer files to touch when the next variant lands. A solution that costs more compute or runs slower is fine if it eliminates a class of code we'd otherwise have to maintain. Lookup tables, hand-coded heuristics, and per-case branches are brittle in a way the bill-of-materials never reflects.
- **Loud breaks.** If an assumption is wrong, the code crashes — it does not silently degrade. No swallowed errors, no fallback chains, no softened tests.
- **Architecture decisions should not accrue tech debt.** A "fine-for-now" choice that the roadmap will force us to unwind in six weeks is a medium-severity finding, not a nit — flag it and propose the seam.
- **DRY and missing-abstraction findings are first-class.** They are not style polish. A missing abstraction today is tomorrow's bug surface × N.

Reviewers — human and automated — weight findings with these values in mind, not just raw severity.

## Fail-Fast / Offensive Programming

Write fail-fast code. If an assumption is wrong, the code crashes — it does not silently degrade.

- Access data directly: `x["key"]`, not `(x or {}).get("key", default)`
- No fallback chains: `a or b or c or default` — pick one source of truth
- No swallowed errors: `try: ... except: pass`
- Avoid `try/except` / `try/catch` in core logic; catch only at system boundaries (user input, external APIs)
- When encountering existing defensive code, rip it out and run tests. If they pass, the defense was bloat.

**Review question:** *Is this code non-defensive? Does it crash loudly when its invariants are violated?*

## Concise Code (LOC is a cost)

Every line earns its place. Verbose code is more bugs, more review time, more drift.

- Don't add `if x:` guards when `x` should always be truthy
- Don't introduce helpers/abstractions for one or two call sites
- Three similar lines > premature abstraction
- When I propose verbose code, push back
- Prefer direct access over fallback chains and nested conditionals
- Every conditional is a maintenance burden — all things equal, avoid it. Each `if`/`else` branch is a state the next reader has to hold and the next change has to update. When you find yourself adding a special case, ask whether a different seam would let you delete it instead.

**Review question:** *Is this concise and elegant, or is there extra code that doesn't pay for itself?*

## Anti-Bloat — Don't Propose These Remedies

LOC is a stand-in for **conditionals + special cases + defensive branches + new abstractions** — those are the actual cost being managed. Each calcified branch survives every future refactor and shapes the next change. The test for any edge-case handler: *does the edge case actually happen, or will it in the near future?* If neither, the remedy is bloat regardless of how the line count nets out.

When reviewing a PR or applying a bot's suggestion, **don't propose** (or accept) these patterns:

- **assert→raise+log replacements outside trust boundaries.** `Fail-Fast` prefers the assertion crashing loudly on internal-caller bugs.
- **isinstance / type-validation checks for internal callers.** Validation belongs at trust boundaries (user input, external APIs); internal calls trust their callers.
- **state-reset / fallback writes** for hypothetical pollution ("set X = default in case prior init left it dirty"). Unless the polluting scenario is observed in production, the fallback locks the surrounding code into preserving both branches forever.
- **wrapper dataclasses, snapshot views, or new DI seams** for one call site. These are remedies for repeated problems; "Three similar lines > premature abstraction" applies.
- **streaming / incremental rewrites** of small in-memory operations on theoretical perf or OOM grounds. Cite a measured failure or skip.
- **companion tests, CI guards, or regression coverage for unreachable scenarios** ("no test for prod re-enable", "no test that this can't happen"). Tests calcify too — a test for a scenario that doesn't occur preserves a contract that may have been wrong, and the next refactor has to keep that contract working forever.

When pushing back on a bot finding whose remedy hits one of these patterns, cite this section by name (`Anti-Bloat`) and apply the LOC-negative or branch-negative version of the fix.

**Review question:** *Does the proposed remedy add a branch / handler / abstraction for a scenario that hasn't been shown to actually occur?*

## Incremental Improvement

Leave each file a little better than you found it. "Better" = fewer LOC, fewer conditionals, fewer special cases, clearer assumptions at the seams it consumes and exposes, and louder failure when those assumptions break. A bit slower at runtime is fine. If you're already editing something and spot an obvious tidy — two nearly-identical mocks that could collapse into a factory, an unused import, a one-line comment that restates the function name — take it.

Split the work: get the business-logic change working and tests passing first, then do the cleanup as a **separate commit**. A reviewer should see the behavioral change cleanly on its own and the cleanup as its own diff.

- Don't block on unrelated cleanup — keep the focused change landable.
- Don't sneak cleanup into the feature commit — it hides the real change.
- If the cleanup is big enough that it deserves its own PR, make it its own PR.

**Review question:** *Did the author leave the file measurably better than they found it, without muddying the business-logic diff?*

## DRY: Find Existing Logic Before Adding New

New functions, user-facing strings, and status reports must be grep'd for near-duplicates before approval. Parallel implementations are the single most common LLM defect.

- Grep the repo for similar function names and near-duplicate strings.
- If a similar implementation exists, unify rather than adding alongside.
- Duplicate logic with an existing repo site: *blocking: dry* — cite the existing site by path.
- Intra-PR duplication (N near-identical blocks in the same diff) is a missing abstraction: *medium: dry*.

**Review question:** *Is this DRY with the rest of the codebase? Are we re-using existing functions where we should?*

## Regression Risk

New code must not quietly change existing business logic. Any change to a function's contract, return shape, side effects, or error behavior needs an explicit callout.

- Changed semantics of an existing function without updating call sites: *blocking: regression*.
- New conditional branch that alters legacy-path behavior: *medium: regression* unless proven safe.
- A bug fix without a regression test that exercises the old-bug path is *blocking: tests*.

**Review question:** *Does this introduce regressions? How does it change existing business logic, and is that change intentional and covered?*

## Reframe the Spec

Any single new block over ~20 LOC is a smell. Pause and ask whether a spec or UX tweak would delete most of that code. Don't silently accept contorted code to match a literal ask — name the simpler alternative so the author can choose.

- If satisfying a spec detail is adding notable complexity: *low: spec-reframe* with the alternative described in one sentence.
- If the complexity is structural (not just a spec ask): escalate to *medium*.

**Review question:** *Would a small change to the spec or UX accomplish the spirit of the request while significantly reducing complexity?*

## Name the Shape

Before writing code, name the *class* of problem: parsing, validation, dispatch, retry, auth, serialization, formatting, batching, state, idempotency, audit logging, feature-flag gating. Each is a recurring shape with a preferred home in the codebase. Pick the shape first; the lines come second.

Two failure modes:

- **Wrong shape.** Regex when the upstream could emit structured data. Hand-rolled validation when pydantic/zod is already in use. `if kind == "A": ... elif kind == "B": ...` instead of a dispatch dict. Bool-soup state instead of an enum. The code works on today's input and silently breaks on the next variant. Regex is the canonical example — treat regex on string-typed input as a smell that *structure got discarded upstream*. The fix is usually to make upstream emit data, not to grow the regex.
- **Five shapes for five similar problems.** Each new feature lands with its own auth check, its own retry loop, its own date format, its own error shape. Each PR looks fine in review; the policy now lives in 17 places. If you are the second instance of a missing shape, *that* is the moment to introduce one — not the fifth.

Before writing: grep for how the codebase already handles this class. If a shape exists, conform. If none exists and you are not the first instance, you have just discovered tech debt — flag it.

**Review question:** *Did the author name the problem class and conform to (or establish) the codebase's shape for it, or did they solve only the literal symptom?*

## Generalize the Fix (Narrow-Fix)

A fix tied to one reported input (regex, string equality, hard-coded special case) is a narrow fix. Ask: what is the next variant of this input, and does this code cover it?

- Narrow fixes that won't survive the obvious next bug are *medium: narrow-fix*.
- If the root cause is systemic (parsing, state machine, schema), fix the class, not the instance.
- `Narrow-Fix` is only valid on the FIRST occurrence of a class on a given PR. A second instance of the same class on the same PR escalates to `Bug-Class-Recurrence` (below) — repeating `Narrow-Fix` review after review erodes the tag's signal and traps the author in a local-fix loop.

**Review question:** *Is this a root-cause fix or a patch for one symptom?*

## Bug-Class-Recurrence

When the same class of bug has been flagged 2+ times on the same PR — across reviews, or across multiple findings within a single review — the right finding is structural, not local. Patching individual instances has reached diminishing returns; the right move is to name the architectural shape that would make the entire class impossible (per-session value types, sealed enums for state, single-owner data, registries, dispatch maps) and recommend that.

A `Bug-Class-Recurrence` finding **replaces** — does not append to — the individual local findings of the same class. Listing both anchors the author on the local fix; they will fix the local one and the structural finding becomes background noise.

If you genuinely cannot name the structural alternative, downgrade to `medium` and surface as an Open Question instead. Do NOT fall back to listing local fixes.

- Same class flagged in 2+ prior reviews → *blocking: bug-class-recurrence*.
- 2+ findings of the same class in the current review draft → *blocking: bug-class-recurrence* (collapse them into one structural finding).
- Supersedes `Narrow-Fix` after the first occurrence of the class.

**Review question:** *Is this the Nth instance of a recurring shape? If so, what's the structural alternative — the one piece of code that would make every instance disappear?*

## Tests

Tests must fail loudly when something is broken. Never add graceful degradation, skips, or soft failures to tests. A broken test should crash, not pass silently.

- Test **user experience and business logic**, not self-evident language/runtime behavior
- A change crossing multiple layers (e.g. Swift → plowd → API) needs 1–2 focused behavior tests, not exhaustive layer-by-layer coverage
- Prefer shared fixtures/factories over inline setup blocks
- Refactor repeated test payloads, DB rows, and lifecycle setup into helpers
- Remove low-value helper tests before adding more boilerplate
- **Run `just test` after every change** — features, fixes, refactors, anything. Don't return to the user with a red bar.
- **`just test` must pass before merging.** The pre-merge gate is non-negotiable: no merge on a red bar, even if the change is "obviously" safe. If a repo doesn't have a `just test`, add one before landing the first PR that needs gating.

New tests in the plow repo should align with `docs/TEST_PATTERNS.md` and `./tests/README.md` — the repo's own test-writing conventions. Tests that diverge from those patterns without a good reason are a *low: tests* finding.

**Review question:** *Are the tests loud-on-failure, aligned with the repo's test patterns, and focused on behavior rather than implementation details?*

## Migrations

Never hand-write Alembic migrations. Always use `alembic revision --autogenerate` to generate them from model changes.

**Review question:** *Was this migration autogenerated?*

---

## How reviewers should weight these

For any PR, reviewers rank findings by severity first (`blocking` → `medium` → `low` → `nit`), then within a severity band by:

1. **Tech-debt and architectural findings** (missing abstraction, DRY violation, design that won't survive the roadmap) — these compound.
2. **Broad-correctness findings** that affect many paths or users.
3. **Surface-area findings** touching many files.
4. **Localized fixes and style** rank LAST within their band.

If two findings compete and one is "code that won't scale as the team grows" vs. one that is "line-level style," the scalability finding wins the higher slot even at the same severity.

## Approval Verdicts

Default verdict is **APPROVE** unless there are `blocking` findings. Medium, low, and nit findings appear in the review but do not prevent approval. Only `blocking` warrants `VERDICT: COMMENT`.

When approving with nit or low items worth noting, use `VERDICT: APPROVE — pending: <item1>, <item2>` so the approval body communicates what minor things would be nice to clean up.

## Pre-Verdict Checklist

Before writing your verdict, for any PR that adds a new function, a new user-facing string, a fix scoped to a single input, or any single new block over ~20 LOC, run these checks inside the checked-out repo:

1. `grep` for near-duplicate function names and strings. Flag hits as *blocking: dry* with the existing path.
2. For fixes scoped to one input, name the obvious next variant and check whether it's covered. If not, *medium: narrow-fix*.
3. If the PR adds complexity to honor a spec detail verbatim, name a UX or contract change that would delete most of it. *low: spec-reframe*.

## Tooling: kid vs grep

Prior-art lookup runs via two paths depending on file type:

- **`*.py`** — semantic search via `kid` (indexes `~/Hacking/plow-kid`, injected into the review as a "PRIOR ART" section). Treat kid hits as leads, not verdicts: each match is either dismissed with a reason (different contract, unavoidable duplication) or raised as a DRY finding.
- **`*.swift`, `*.ts`, `*.tsx`, everything else** — fall back to `grep` inside the checked-out repo for function names and near-duplicate strings (Pre-Verdict Checklist step 1). Kid Swift support is a planned follow-up; until then, do not assume kid covers Phoenix or frontend code.

Do not silently trust "no kid hits = DRY-clean". Confirm which path applied.

A reviewer should have caught the third divergent copy on arrival. That is what "grep before you approve" looks like in practice.

## Core Philosophy

- Follow red -> green -> refactor TDD: write the failing test first, implement
  the smallest change that passes, then clean up while tests stay green.

  if this failed?
- Keep setup compact and readable. If a helper does not reduce real repetition
  or clarify the test's intent, do not add it.

     2	
     3	Last updated: 2026-03-03
     4	
     5	## Purpose
     6	
     7	This document describes:
     8	1. The architecture Phoenix is running today.

    12

    14	
    15	## System Overview (Current)
    16	
    17	Core components:
    18	1. `Phoenix.app` (macOS UI client).
    19	2. `phoenixd` (local policy and orchestration daemon).
    20	3. `openclaw-gateway` (tool runtime and agent loop, `exec.host = "gateway"` inside Alpine VM).
    21	4. `phoenix-gateway-guard` OpenClaw plugin (`before_tool_call` / `after_tool_call` hooks).
    22	
    23	Current runtime shape:
    24	1. Gateway is bound to loopback (`127.0.0.1`) with token auth.
    25	2. PhoenixD manages OpenClaw config reconciliation and process supervision.
    26	3. PhoenixD resolves exec approval events over gateway operator websocket.
    27	4. Tool policy decisions are requested from PhoenixD by plugin hook callbacks.
    28	
    29	## Trust Boundaries (Current)
    30	
    31	1. UI does not need direct gateway token by design.
    32	2. Gateway token is stored in user home (`~/.phoenix/openclaw/secrets/gateway-token`) with mode `0600`.
    33	3. Gateway/API access is local-only by bind setting, but same-user processes remain in the trust boundary.
    34	4. OpenClaw runtime (gateway + node host) is currently trusted to honor plugin/approval contracts.
    35	
    36	## Request Flow (Current)
    37	
    38	Non-exec tool call:
    39	1. OpenClaw gateway receives tool request.
    40	2. Plugin `before_tool_call` calls PhoenixD `/internal/gateway/policy/tool-check`.
    41	3. If denied or unavailable, plugin blocks (fail closed).
    42	4. If allowed, OpenClaw gateway executes the tool.
    43	5. Plugin `after_tool_call` posts result to PhoenixD audit endpoint.
    44	
    45	Exec tool call:
    46	1. Same `before_tool_call` policy gate applies.
    47	2. Gateway raises `exec.approval.requested`.
    48	3. PhoenixD operator client approves or denies.
    49	4. If approved, execution happens via OpenClaw node host (`system.run`), not inside PhoenixD process.
    50	
    51	## What Is Secure Today
    52	
    53	1. Default local exposure is reduced (`loopback` + token auth).
    54	2. Policy checks for tool calls are centralized in PhoenixD (plugin gate).
    55	3. Exec has a second approval layer (`ask=always`, fallback deny behavior).
    56	4. PhoenixD owns config reconciliation and can enforce baseline guardrails.
    57	5. Operational audit trail exists for tool checks/results and process status.
    58	
    59	## What Is Not Guaranteed Yet
    60	
    61	1. Full runtime isolation from OpenClaw bugs is not guaranteed.
    62	2. Gateway network egress is not fully forced through Phoenix today.
    63	3. Same-user token theft is still possible on a compromised local account.
    64	4. OpenClaw runtime remains a trusted enforcement participant in current model.
    65	5. OS-level sandbox boundaries are not yet applied to gateway/node runtime paths.
    66	6. `agentId` is not currently a cryptographically attested identity primitive and should not be used as a sole authorization key.
    67	7. Plugin->Phoenix policy callbacks (`/internal/gateway/policy/*`) are local but currently do not have an additional dedicated auth secret/signature layer.
    68	
    69	## Principal Risks
    70	
    71	1. OpenClaw bypass bug risk:
    72	   - A severe runtime bug could bypass plugin/approval paths.
    73	2. Gateway direct egress risk:
    74	   - Gateway may call model/provider/channels directly unless egress is constrained.
    75	3. Same-user token risk:
    76	   - Any process running as the same macOS user can potentially read the gateway token file.
    77	4. Control-plane surface risk:
    78	   - Any client with token and loopback access can attempt privileged gateway operations.
    79	5. Identity confusion risk:
    80	   - Policy decisions based only on payload-level `agentId` can be weaker than decisions keyed by stronger session/channel context.
    81	
    82	## Identity Signals and Trust Strength (Current)
    83	
    84	Signals currently observed in exec approval payloads include `agentId`, `sessionKey`, `nodeId`, and `systemRunBindingV1`.

    86	Current trust guidance:
    87	1. Treat `agentId` as a UX/debug attribute, not the sole security principal.
    88	2. Treat `(nodeId, sessionKey)` as the primary policy key in current architecture.
    89	3. Enforce consistency checks across duplicate fields (for example, `request.sessionKey` and `request.systemRunBindingV1.sessionKey`) and deny on mismatch.
    90	4. Preserve fail-closed behavior for ambiguous or missing identity fields.
    91	
    92	## Hardening Roadmap
    93	
    94	### Phase 0: Baseline discipline (now)
    95	
    96	1. Keep loopback bind and token auth mandatory.
    97	2. Keep plugin required and fail-closed for policy check.
    98	3. Keep exec ask/approval as deny-by-default fallback.
    99	4. Keep OpenClaw version pinning and deterministic config reconciliation.
   100	5. Prefer policy mode `ask` after approval UI is live.
   101	
   102	### Phase 1: UX and approval authority hardening
   103	
   104	1. Add PhoenixD <-> app websocket channel for real-time approvals.
   105	2. Move default approval mode from auto-allow to ask.
   106	3. Improve retry behavior around first exec/pairing race.
   107	4. Add token rotation command and operational runbook.
   108	5. Add explicit principal mapping and mismatch-deny logic keyed by `(nodeId, sessionKey)`.
   109	
   110	### Phase 2: Packaging and deterministic runtime
   111	
   112	1. Bundle pinned Node + OpenClaw runtime in signed installer payload.
   113	2. Remove end-user npm/pnpm install dependencies.
   114	3. Ensure daemon startup and bootstrap are deterministic on fresh machine.
   115	4. Migrate startup ownership to launchd/SMAppService model.
   116	5. Add plugin->Phoenix policy endpoint authentication (shared secret and/or signed request envelope).
   117	
   118	### Phase 3: Execution isolation
   119	
   120	1. Introduce Phoenix-managed sandboxed executor helper for exec path.
   121	2. Route approved exec through sandbox helper instead of broad host runtime.
   122	3. Enforce file/network/process limits at OS sandbox boundary.
   123	
   124	### Phase 4: Egress and runtime containment
   125	
   126	1. Enforce proxy-only provider/network path where gateway cannot bypass Phoenix proxy.
   127	2. Add strong egress controls for gateway runtime.
   128	3. Move toward full gateway/node isolation (containerized boundary or VM).
   129	4. Optionally adopt custom Phoenix node runtime if protocol maintenance cost is acceptable.
   130	
   131	## Strategic Options and Tradeoffs
   132	
   133	Option A: Keep official OpenClaw node host + sandboxed exec helper
   134	1. Lower maintenance cost.
   135	2. Faster path to meaningful hardening.
   136	3. Still trusts OpenClaw for non-exec behavior unless further containment is added.
   137	
   138	Option B: Build custom Phoenix node runtime
   139	1. Stronger control over exec enforcement.
   140	2. Higher long-term protocol compatibility and maintenance burden.
   141	3. Best used after installer and sandbox foundations are stable.
   142	
   143	Option C: Full runtime containment (VM-level)
   144	1. Strongest isolation model.
   145	2. Highest complexity and operational cost.
   146	3. Appropriate for high-assurance security target.
   147	
   148	## Decision Guidance
   149	
   150	Recommended near-term order:
   151	1. Finish approval websocket/UI path.
   152	2. Ship deterministic installer/runtime packaging.
   153	3. Implement sandboxed exec helper.
   154	4. Add proxy-only egress enforcement.
   155	5. Reassess need for custom node runtime after measuring residual risk.

   158	
   159	1. What minimum security bar is required for first public installer release?
   160	2. Is same-user token storage acceptable until Keychain migration, or should Keychain be mandatory before release?
   161	3. Do we treat non-exec gateway egress lock-down as release-blocking or phase 2 hardening?
   162	4. At what point does custom node runtime ROI exceed maintenance cost?

    45	
    46	After validation, a deeper question reframed the productization conversation: should the agent ever see a credential index, and should every credential disclosure require explicit human approval? The desktop app's "Authorization for command-line use" dropdown almost certainly has no sub-minute setting. Shipping a `signout`-before-fetch hack would risk surprising the user with re-prompts in their other terminals and still wouldn't deliver a deterministic window. Re-evaluate when 1P ships finer-grained auth control or when we move to Path C.
    53	
    54	**Path B — move URL→item match server-side.** Broker takes a URL, returns one matching credential or null. `onepassword_list_logins` disappears from the agent tool surface. Eliminates **index disclosure** — arguably the larger of the two privacy concerns since the index reveals which sites the user has accounts on (metadata often as sensitive as the credentials themselves). Agent still sees plaintext for the one fetched credential. Estimated 1–2 weeks: URL-match logic in `onepassword.py`, retire the index tool, update SKILL.md to a single-step "give the broker a URL" pattern. Largest privacy win for the engineering investment.

    56	**Path C — match the full 1P-shape: never put credentials in agent context.** Both URL match AND form-fill move out of the agent's tool path. Openclaw's browser-fill primitive (or a native bridge) requests a disclosure from a host-side service; user approves; credential is injected into browser DOM without passing through the LLM. Requires extending openclaw's browser-fill, building a native consent surface, and a trust contract between broker and browser instance. Weeks of work. The right end-state, but probably not the immediate next step.
    57	
    58	**Path D — engage 1Password directly.** EAP language suggests they're soliciting integrators. If they open the Agentic Autofill protocol to third-party agent runtimes, Plow could become a Browserbase peer rather than reinventing the consent surface. Long lead time; lowest engineering tax if it works out. Worth pursuing in parallel as a follow-up — the answer materially affects whether B/C are necessary.
    59	
    60	Path A is blocked at the `op` v2 layer; revisit when 1P exposes finer-grained auth control. Path B is the highest-value architectural move for the cost — index disclosure is the bigger of the two privacy gaps. Path C is the right end-state but probably not next. Path D is worth pursuing in parallel.
