### Finding 1 — blocking
This branch contains a real bug fix (`acquire` failure now rolls the toggle back OFF), but there is still no regression test for that path or the adjacent state transitions it depends on. `KeepMacAwake` now has persisted-on init, successful enable, failed enable with rollback, and quit-time release behavior, while both `SettingsView` and `StatusView` bind directly to `isEnabled`. With no automated coverage, the exact bug this PR fixed can regress silently and leave users with a toggle that lies about whether their Mac will stay awake. The cheapest seam is injecting an assertion client plus `UserDefaults` suite into `KeepMacAwake` and unit-testing those transitions.

### Finding 2 — blocking
The installer-sizing work also landed without regression coverage even though this branch already needed two follow-up fix commits for it. `InstallerView` now relies on measured content height, measured overlay height, and a screen-height cap to keep the new settings rows from being hidden under `DownloadBarView` or pushed off-screen, but no test pins any of those branches. That leaves the author’s own test-plan cases manual-only. A low-bloat seam is to extract the height math into a pure helper like `targetWindowHeight(screen:contentHeight:overlayHeight:visibleFrameHeight:)` and unit-test the default, overlay-visible, and capped cases; the existing source-assertion file can keep the preference-key wiring honest.
