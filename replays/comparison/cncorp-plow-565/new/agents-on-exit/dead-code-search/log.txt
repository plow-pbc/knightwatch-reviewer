# Plow

Plow is an agent platform that gives OpenClaw agents narrow, permission-gated access to cloud services (Gmail, Slack) and on-device capabilities through a local approval layer.

The system has three tiers:

- **Plow API** (`api/`) — cloud service that owns connector OAuth, opportunity matching, marketplace skills, and policy enforcement
- **Plow app** (`app/`) — macOS app (Swift) + plowd daemon (Python) that manages on-device approval, gateway configuration, and VM lifecycle
- **Plow CLI** (`cli/`) — narrow tool surface inside the OpenClaw container, talks directly to the Plow API

# Plow App

The Plow macOS app is currently a local OpenClaw wrapper/orchestrator centered on **PlowD** (the daemon).

## Current Focus

As of this branch, the primary path is:

- PlowD starts and manages OpenClaw gateway + node host locally.
- Plow app tooling keeps app-managed config/state under `~/Library/Application Support/co.plow.app/...` instead of `~/.openclaw/...`.
- Plow gateway policy/operator logic handles pairing and tool approval flows.

The older Wren/Inbox agent-first flow is not the main focus right now, and agent registration on daemon startup is currently disabled.

## Architecture (Now)

- **PlowD (FastAPI daemon)**: process manager + policy service + status API.
- **OpenClaw gateway**: local websocket gateway (`127.0.0.1`, token auth).
- **OpenClaw node host**: execution host process paired to the gateway.
- **OpenClaw dashboard/web chat**: optional local UI client that connects to the same gateway.
- **Plow macOS app**: still exists as iMessage bridge/UI layer, but not the center of this PR.

OpenClaw terms vs app terms:

- PlowD == daemon/operator-side process (not the gateway binary itself)
- OpenClaw gateway == websocket service PlowD launches/manages
- Web client/dashboard == separate operator client that can connect locally when paired/authenticated

State roots are bundle-scoped:

- production app: `~/Library/Application Support/co.plow.app/`
- default dev app: `~/Library/Application Support/co.plow.app.dev/`
- worktree/dev variants: `~/Library/Application Support/<bundle-id>/`

# Phoenix + OpenClaw Security Architecture (Current State and Hardening Roadmap)

Last updated: 2026-03-03

## Purpose

This document describes:
1. The architecture Phoenix is running today.
2. What security guarantees exist right now.
3. What is not guaranteed yet.
4. A phased hardening roadmap.

This is a living architecture document for implementation and product/security decisions.

## System Overview (Current)

Core components:
1. `Phoenix.app` (macOS UI client).
2. `phoenixd` (local policy and orchestration daemon).
3. `openclaw-gateway` (tool runtime and agent loop, `exec.host = "gateway"` inside Alpine VM).
4. `phoenix-gateway-guard` OpenClaw plugin (`before_tool_call` / `after_tool_call` hooks).

Current runtime shape:
1. Gateway is bound to loopback (`127.0.0.1`) with token auth.
2. PhoenixD manages OpenClaw config reconciliation and process supervision.
3. PhoenixD resolves exec approval events over gateway operator websocket.
4. Tool policy decisions are requested from PhoenixD by plugin hook callbacks.

## Trust Boundaries (Current)

1. UI does not need direct gateway token by design.
2. Gateway token is stored in user home (`~/.phoenix/openclaw/secrets/gateway-token`) with mode `0600`.
3. Gateway/API access is local-only by bind setting, but same-user processes remain in the trust boundary.
4. OpenClaw runtime (gateway + node host) is currently trusted to honor plugin/approval contracts.

## Request Flow (Current)

Non-exec tool call:
1. OpenClaw gateway receives tool request.
2. Plugin `before_tool_call` calls PhoenixD `/internal/gateway/policy/tool-check`.
3. If denied or unavailable, plugin blocks (fail closed).
4. If allowed, OpenClaw gateway executes the tool.
5. Plugin `after_tool_call` posts result to PhoenixD audit endpoint.

Exec tool call:
1. Same `before_tool_call` policy gate applies.
2. Gateway raises `exec.approval.requested`.
3. PhoenixD operator client approves or denies.
4. If approved, execution happens via OpenClaw node host (`system.run`), not inside PhoenixD process.

## What Is Secure Today

1. Default local exposure is reduced (`loopback` + token auth).
2. Policy checks for tool calls are centralized in PhoenixD (plugin gate).
3. Exec has a second approval layer (`ask=always`, fallback deny behavior).
4. PhoenixD owns config reconciliation and can enforce baseline guardrails.
5. Operational audit trail exists for tool checks/results and process status.

## What Is Not Guaranteed Yet

1. Full runtime isolation from OpenClaw bugs is not guaranteed.
2. Gateway network egress is not fully forced through Phoenix today.
3. Same-user token theft is still possible on a compromised local account.
4. OpenClaw runtime remains a trusted enforcement participant in current model.
5. OS-level sandbox boundaries are not yet applied to gateway/node runtime paths.
6. `agentId` is not currently a cryptographically attested identity primitive and should not be used as a sole authorization key.
7. Plugin->Phoenix policy callbacks (`/internal/gateway/policy/*`) are local but currently do not have an additional dedicated auth secret/signature layer.

## Principal Risks

1. OpenClaw bypass bug risk:
   - A severe runtime bug could bypass plugin/approval paths.
2. Gateway direct egress risk:
   - Gateway may call model/provider/channels directly unless egress is constrained.
3. Same-user token risk:
   - Any process running as the same macOS user can potentially read the gateway token file.
4. Control-plane surface risk:
   - Any client with token and loopback access can attempt privileged gateway operations.
5. Identity confusion risk:
   - Policy decisions based only on payload-level `agentId` can be weaker than decisions keyed by stronger session/channel context.

## Identity Signals and Trust Strength (Current)

Signals currently observed in exec approval payloads include `agentId`, `sessionKey`, `nodeId`, and `systemRunBindingV1`.

Current trust guidance:
1. Treat `agentId` as a UX/debug attribute, not the sole security principal.
2. Treat `(nodeId, sessionKey)` as the primary policy key in current architecture.
3. Enforce consistency checks across duplicate fields (for example, `request.sessionKey` and `request.systemRunBindingV1.sessionKey`) and deny on mismatch.
4. Preserve fail-closed behavior for ambiguous or missing identity fields.

## Hardening Roadmap

### Phase 0: Baseline discipline (now)

1. Keep loopback bind and token auth mandatory.
2. Keep plugin required and fail-closed for policy check.
3. Keep exec ask/approval as deny-by-default fallback.
4. Keep OpenClaw version pinning and deterministic config reconciliation.
5. Prefer policy mode `ask` after approval UI is live.

### Phase 1: UX and approval authority hardening

1. Add PhoenixD <-> app websocket channel for real-time approvals.
2. Move default approval mode from auto-allow to ask.
3. Improve retry behavior around first exec/pairing race.
4. Add token rotation command and operational runbook.
5. Add explicit principal mapping and mismatch-deny logic keyed by `(nodeId, sessionKey)`.

### Phase 2: Packaging and deterministic runtime

1. Bundle pinned Node + OpenClaw runtime in signed installer payload.
2. Remove end-user npm/pnpm install dependencies.
3. Ensure daemon startup and bootstrap are deterministic on fresh machine.
4. Migrate startup ownership to launchd/SMAppService model.
5. Add plugin->Phoenix policy endpoint authentication (shared secret and/or signed request envelope).

### Phase 3: Execution isolation

1. Introduce Phoenix-managed sandboxed executor helper for exec path.
2. Route approved exec through sandbox helper instead of broad host runtime.
3. Enforce file/network/process limits at OS sandbox boundary.

### Phase 4: Egress and runtime containment

1. Enforce proxy-only provider/network path where gateway cannot bypass Phoenix proxy.
2. Add strong egress controls for gateway runtime.
3. Move toward full gateway/node isolation (containerized boundary or VM).
4. Optionally adopt custom Phoenix node runtime if protocol maintenance cost is acceptable.

## Strategic Options and Tradeoffs

Option A: Keep official OpenClaw node host + sandboxed exec helper
1. Lower maintenance cost.
2. Faster path to meaningful hardening.
3. Still trusts OpenClaw for non-exec behavior unless further containment is added.

Option B: Build custom Phoenix node runtime
1. Stronger control over exec enforcement.
2. Higher long-term protocol compatibility and maintenance burden.
3. Best used after installer and sandbox foundations are stable.

Option C: Full runtime containment (VM-level)
1. Strongest isolation model.
2. Highest complexity and operational cost.
3. Appropriate for high-assurance security target.

## Decision Guidance

Recommended near-term order:
1. Finish approval websocket/UI path.
2. Ship deterministic installer/runtime packaging.
3. Implement sandboxed exec helper.
4. Add proxy-only egress enforcement.
5. Reassess need for custom node runtime after measuring residual risk.

1. What minimum security bar is required for first public installer release?
2. Is same-user token storage acceptable until Keychain migration, or should Keychain be mandatory before release?
3. Do we treat non-exec gateway egress lock-down as release-blocking or phase 2 hardening?
4. At what point does custom node runtime ROI exceed maintenance cost?
