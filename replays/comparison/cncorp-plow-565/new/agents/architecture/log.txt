# Plow

Plow is an agent platform that gives OpenClaw agents narrow, permission-gated access to cloud services (Gmail, Slack) and on-device capabilities through a local approval layer.

The system has three tiers:

- **Plow API** (`api/`) — cloud service that owns connector OAuth, opportunity matching, marketplace skills, and policy enforcement
- **Plow app** (`app/`) — macOS app (Swift) + plowd daemon (Python) that manages on-device approval, gateway configuration, and VM lifecycle
- **Plow CLI** (`cli/`) — narrow tool surface inside the OpenClaw container, talks directly to the Plow API

## Architecture

```mermaid
flowchart TB
    User[User]

    subgraph Mac["macOS Host"]
        Phoenix["Plow.app\n(Swift · Virtualization.framework)"]
        Plowd["plowd daemon\n(:18788 · Python FastAPI)"]

        subgraph VM["Alpine Linux VM"]
            GW["OpenClaw Gateway\n(:18789)"]
            Chan["plow-imessage channel"]
            OC["OpenClaw Agent"]
            CLI["plow CLI\n(containercli)"]
        end
    end

    subgraph Cloud["Cloud (AWS ECS)"]
        API["Plow API\n(FastAPI)"]
        DB["PostgreSQL\n+ pgvector"]
        OE["Opportunity Engine\n(LLM-based JTBD matching)"]

        subgraph Connectors
            GM[Gmail]
            SL[Slack]
        end

        subgraph Channels

        end
    end

    User --> Phoenix
    Phoenix -->|vsock| VM
    Plowd -->|vsock| VM
    Plowd -->|"gateway operator\n(WebSocket)"| GW
```

**Data flow:**

- Inbound SMS → Linq → Plow API webhook → WebSocket → app → LLM → outbound SMS ([full routing doc](docs/architecture/sms-inbound-routing.md))
- `plow gmail search "mortgage"` → CLI → Plow API → Gmail
- `plow slack search "renewal"` → CLI → Plow API → Slack
- OpenClaw never holds raw OAuth tokens — it sees narrow CLI tools only

# Plow App

The Plow macOS app is currently a local OpenClaw wrapper/orchestrator centered on **PlowD** (the daemon).

## Current Focus

As of this branch, the primary path is:

- PlowD starts and manages OpenClaw gateway + node host locally.
- Plow app tooling keeps app-managed config/state under `~/Library/Application Support/co.plow.app/...` instead of `~/.openclaw/...`.
- Plow gateway policy/operator logic handles pairing and tool approval flows.

The older Wren/Inbox agent-first flow is not the main focus right now, and agent registration on daemon startup is currently disabled.

## Architecture (Now)

- **PlowD (FastAPI daemon)**: process manager + policy service + status API.
- **OpenClaw gateway**: local websocket gateway (`127.0.0.1`, token auth).
- **OpenClaw node host**: execution host process paired to the gateway.
- **OpenClaw dashboard/web chat**: optional local UI client that connects to the same gateway.
- **Plow macOS app**: still exists as iMessage bridge/UI layer, but not the center of this PR.

OpenClaw terms vs app terms:

- PlowD == daemon/operator-side process (not the gateway binary itself)
- OpenClaw gateway == websocket service PlowD launches/manages
- Web client/dashboard == separate operator client that can connect locally when paired/authenticated

State roots are bundle-scoped:

- production app: `~/Library/Application Support/co.plow.app/`
- default dev app: `~/Library/Application Support/co.plow.app.dev/`
- worktree/dev variants: `~/Library/Application Support/<bundle-id>/`

# OpenClaw + Phoenix Architecture

*Last updated: 2026-03-04*

> **Historical.** This document describes the pre-Lima architecture: a custom Alpine
> rootfs booted via `Virtualization.framework` with vsock/socat bridges. On branch
> `vm/lima-migration`, OpenClaw now runs as an OCI container inside a Lima-managed
> Debian guest (`app/vm/lima.yaml`, `app/Phoenix/LimaVMManager.swift`), and the
> rootfs/initramfs/vsock machinery below is gone. Gatekeeper (Camoufox) still uses
> `Virtualization.framework` on this branch. See PR #310 for the Lima migration
> (openclaw + Debian guest) and the camoufox-on-lima follow-up.

---

## Ownership Map

Every component in this system is owned by exactly one of two codebases:

| Owner | Codebase | Language | What It Owns |
|-------|----------|----------|-------------|
| **OpenClaw** (upstream) | `openclaw/upstream` | Node.js / TypeScript | Gateway, Plugin SDK, Channels, Heartbeat, Cron, ACP, Sessions, Control UI, Pi agent runner |
| **Phoenix** (ours) | `app/`, `app/plowd/`, `app/vm/` | Python + Swift + JS plugins | PhoenixD daemon, macOS app, VM image, phoenix-gateway-guard plugin, plow-imessage channel plugin, config generation |

**Phoenix owns zero lines of the Gateway.** We treat OpenClaw as an opaque
runtime binary. Our integration surface is:

1. **VM management** — we run `openclaw gateway run` inside an Alpine Linux VM (Virtualization.framework)
2. **WebSocket operator protocol** — we connect as an operator to observe and approve
3. **Plugin hooks** — our JS plugin runs inside their Gateway process
4. **Channel plugin** — our plow-imessage channel plugin runs inside their Gateway process
5. **Config generation** — we write `openclaw.json` with locked guardrails

**Note:** We use `exec.host = "gateway"` — the Gateway executes commands directly
(no separate Node Host process). The VM itself provides sandboxing.

## VM Sandbox Architecture

### Why a VM?

- **Isolation**: The Gateway executes arbitrary shell commands on behalf of agents
  (`exec.host = "gateway"`). Running inside a VM limits blast radius — a compromised
  agent cannot access the host filesystem, keychain, or other processes.
- **Reproducibility**: The Alpine rootfs is built deterministically via Docker. Every
  install gets the same sandbox environment.
- **Transparency**: vsock + socat bridges make the VM boundary invisible to both
  PhoenixD and OpenClaw. Gateway still binds `127.0.0.1:18789`, PhoenixD still connects
  to that port — the bridges handle the translation.

### Boot Chain

```
VZLinuxBootLoader (direct kernel boot, no UEFI/bootloader)
  vmlinuz (Alpine linux-virt kernel)
  initramfs.img (custom, static busybox)
    → load kernel modules: virtio, ext4, vsock, virtiofs, virtio_net
    → mount /dev/vda (ext4 rootfs)
    → switch_root

  /sbin/init (PID 1, shell script)
    → mount /proc, /sys, /dev
    → ip link set lo up
    → bring up NAT interface (eth0) via DHCP
    → set deterministic DNS (8.8.8.8, 1.1.1.1)
    → mount virtiofs shares
    → bind-mount /vm-logs → /tmp/openclaw (shared log access)
    → start socat bridges (A + B)
    → signal host readiness via vsock :18787
    → wait for PhoenixD to seed /config/gateway/openclaw.json + gateway token
    → start openclaw-start.sh (gateway)
    → wait for gateway readiness on :18789 (up to 30s)
    → wait (PID 1 keeps running)
```

### Bridge Architecture

Four bridges connect host and guest. Guest-side bridges are socat (in init script).
Host-side bridges are Swift (in VMManager.swift) using NWListener and VZVirtioSocketDevice.

```
HOST (macOS)                                 VM (Alpine Linux)

Bridge C (Swift, VMManager)                  Bridge A (socat, init)
  vsock :18788 listener ◄────vsock────►  TCP-LISTEN:18788 → VSOCK-CONNECT:2:18788
  → relay to TCP 127.0.0.1:18788             (plugin HTTP → PhoenixD)

Bridge D (Swift, VMManager)                  Bridge B (socat, init)
  NWListener TCP :18789 ────vsock────►  VSOCK-LISTEN:18789 → TCP:127.0.0.1:18789
  → relay to vsock :18789                    (PhoenixD → Gateway)
```

**Traffic flows:**
1. PhoenixD (host :18788) ← Bridge C ← vsock ← Bridge A ← plugin HTTP (guest)
2. PhoenixD (host) → TCP :18789 → Bridge D → vsock → Bridge B → Gateway :18789 (guest)

Port 18787 is used only for the one-shot readiness signal (guest → host).

### VM Networking

The VM has outbound internet via `VZNATNetworkDeviceAttachment` (Apple's built-in NAT).
This is needed for channel plugins (e.g., plow-imessage WebSocket to `api.plow.co`).

**Requirements for outbound to work:**
- `virtio_net` kernel module loaded in initramfs (plus `failover`, `net_failover`)
- NAT interface brought up via `ip link set eth0 up` + `udhcpc` in init
- Deterministic DNS written to `/etc/resolv.conf` (DHCP DNS is unreliable)
- `ca-certificates` installed in the rootfs (for HTTPS)

### virtiofs Mounts

| Tag | Guest Path | Mode | Purpose |
|-----|-----------|------|---------|
| workspace | /workspace | rw | Agent working directory |
| config | /config | rw | Gateway config, secrets, sessions, agents |
| mounts | /mounts | rw | Additional host paths |
| plugin | /plugin | ro | phoenix-gateway-guard plugin (from app bundle) |
| channel | /plow-imessage | ro | plow-imessage channel plugin (from app bundle) |
| vm-logs | /vm-logs | rw | Shared log directory (bind-mounted to /tmp/openclaw) |
| desktop | /host/Desktop | rw | Host user's Desktop (when mounted) |

Plugin and channel dirs are compiled JS from the app bundle, mounted read-only.
Config must be rw — the Gateway writes `openclaw.json`, sessions, and secrets.
Host-share paths are strict: `/host/<Name>` exists only when the corresponding
share is actually mounted. The guest does not keep placeholder directories for
absent shares. Updating the share set requires a VM restart.
Phoenix JSON-encodes the active `[HostShare]` list into the
`PLOWD_OPENCLAW_ACTIVE_HOST_SHARES` environment variable before spawning
plowd; plowd reads that env var as its sole source of truth for which
host shares are live this session.

### rootfs Management

The rootfs image is copied from the signed app bundle to `~/.phoenix/vm/rootfs.img`
(writable location required by Virtualization.framework). VMManager auto-detects
bundle changes by comparing file sizes — if the bundle rootfs differs from the local
copy, it replaces the local copy on next boot. No manual deletion needed.

### VM Image Build

3-stage Dockerfile in `app/vm/build-vm-image`:

1. **builder** — Alpine + Node.js + `npm install -g openclaw@<version>`
2. **runtime** — Clean Alpine with `linux-virt` kernel, socat, Node.js, ca-certificates.
   Copies node_modules from builder. Copies init scripts.
3. **initramfs-builder** — Builds custom initramfs from runtime's kernel modules
   (static busybox, decompressed `.ko` files, custom init script)

Outputs: `app/vm/output/{vmlinuz, initramfs.img, rootfs.img}`

**Why custom initramfs?** Alpine's stock `initramfs-virt` uses OpenRC — incompatible
with Virtualization.framework's direct kernel boot. Our initramfs uses static busybox
and loads only the modules we need.

# Phoenix + OpenClaw Security Architecture (Current State and Hardening Roadmap)

Last updated: 2026-03-03

## Purpose

This document describes:
1. The architecture Phoenix is running today.
2. What security guarantees exist right now.
3. What is not guaranteed yet.
4. A phased hardening roadmap.

This is a living architecture document for implementation and product/security decisions.

## System Overview (Current)

Core components:
1. `Phoenix.app` (macOS UI client).
2. `phoenixd` (local policy and orchestration daemon).
3. `openclaw-gateway` (tool runtime and agent loop, `exec.host = "gateway"` inside Alpine VM).
4. `phoenix-gateway-guard` OpenClaw plugin (`before_tool_call` / `after_tool_call` hooks).

Current runtime shape:
1. Gateway is bound to loopback (`127.0.0.1`) with token auth.
2. PhoenixD manages OpenClaw config reconciliation and process supervision.
3. PhoenixD resolves exec approval events over gateway operator websocket.
4. Tool policy decisions are requested from PhoenixD by plugin hook callbacks.

## Trust Boundaries (Current)

1. UI does not need direct gateway token by design.
2. Gateway token is stored in user home (`~/.phoenix/openclaw/secrets/gateway-token`) with mode `0600`.
3. Gateway/API access is local-only by bind setting, but same-user processes remain in the trust boundary.
4. OpenClaw runtime (gateway + node host) is currently trusted to honor plugin/approval contracts.

## Request Flow (Current)

Non-exec tool call:
1. OpenClaw gateway receives tool request.
2. Plugin `before_tool_call` calls PhoenixD `/internal/gateway/policy/tool-check`.
3. If denied or unavailable, plugin blocks (fail closed).
4. If allowed, OpenClaw gateway executes the tool.
5. Plugin `after_tool_call` posts result to PhoenixD audit endpoint.

Exec tool call:
1. Same `before_tool_call` policy gate applies.
2. Gateway raises `exec.approval.requested`.
3. PhoenixD operator client approves or denies.
4. If approved, execution happens via OpenClaw node host (`system.run`), not inside PhoenixD process.

## What Is Secure Today

1. Default local exposure is reduced (`loopback` + token auth).
2. Policy checks for tool calls are centralized in PhoenixD (plugin gate).
3. Exec has a second approval layer (`ask=always`, fallback deny behavior).
4. PhoenixD owns config reconciliation and can enforce baseline guardrails.
5. Operational audit trail exists for tool checks/results and process status.

## What Is Not Guaranteed Yet

1. Full runtime isolation from OpenClaw bugs is not guaranteed.
2. Gateway network egress is not fully forced through Phoenix today.
3. Same-user token theft is still possible on a compromised local account.
4. OpenClaw runtime remains a trusted enforcement participant in current model.
5. OS-level sandbox boundaries are not yet applied to gateway/node runtime paths.
6. `agentId` is not currently a cryptographically attested identity primitive and should not be used as a sole authorization key.
7. Plugin->Phoenix policy callbacks (`/internal/gateway/policy/*`) are local but currently do not have an additional dedicated auth secret/signature layer.

## Principal Risks

1. OpenClaw bypass bug risk:
   - A severe runtime bug could bypass plugin/approval paths.
2. Gateway direct egress risk:
   - Gateway may call model/provider/channels directly unless egress is constrained.
3. Same-user token risk:
   - Any process running as the same macOS user can potentially read the gateway token file.
4. Control-plane surface risk:
   - Any client with token and loopback access can attempt privileged gateway operations.
5. Identity confusion risk:
   - Policy decisions based only on payload-level `agentId` can be weaker than decisions keyed by stronger session/channel context.

## Identity Signals and Trust Strength (Current)

Signals currently observed in exec approval payloads include `agentId`, `sessionKey`, `nodeId`, and `systemRunBindingV1`.

Current trust guidance:
1. Treat `agentId` as a UX/debug attribute, not the sole security principal.
2. Treat `(nodeId, sessionKey)` as the primary policy key in current architecture.
3. Enforce consistency checks across duplicate fields (for example, `request.sessionKey` and `request.systemRunBindingV1.sessionKey`) and deny on mismatch.
4. Preserve fail-closed behavior for ambiguous or missing identity fields.

## Hardening Roadmap

### Phase 0: Baseline discipline (now)

1. Keep loopback bind and token auth mandatory.
2. Keep plugin required and fail-closed for policy check.
3. Keep exec ask/approval as deny-by-default fallback.
4. Keep OpenClaw version pinning and deterministic config reconciliation.
5. Prefer policy mode `ask` after approval UI is live.

### Phase 1: UX and approval authority hardening

1. Add PhoenixD <-> app websocket channel for real-time approvals.
2. Move default approval mode from auto-allow to ask.
3. Improve retry behavior around first exec/pairing race.
4. Add token rotation command and operational runbook.
5. Add explicit principal mapping and mismatch-deny logic keyed by `(nodeId, sessionKey)`.

### Phase 2: Packaging and deterministic runtime

1. Bundle pinned Node + OpenClaw runtime in signed installer payload.
2. Remove end-user npm/pnpm install dependencies.
3. Ensure daemon startup and bootstrap are deterministic on fresh machine.
4. Migrate startup ownership to launchd/SMAppService model.
5. Add plugin->Phoenix policy endpoint authentication (shared secret and/or signed request envelope).

### Phase 3: Execution isolation

1. Introduce Phoenix-managed sandboxed executor helper for exec path.
2. Route approved exec through sandbox helper instead of broad host runtime.
3. Enforce file/network/process limits at OS sandbox boundary.

### Phase 4: Egress and runtime containment

1. Enforce proxy-only provider/network path where gateway cannot bypass Phoenix proxy.
2. Add strong egress controls for gateway runtime.
3. Move toward full gateway/node isolation (containerized boundary or VM).
4. Optionally adopt custom Phoenix node runtime if protocol maintenance cost is acceptable.

## Strategic Options and Tradeoffs

Option A: Keep official OpenClaw node host + sandboxed exec helper
1. Lower maintenance cost.
2. Faster path to meaningful hardening.
3. Still trusts OpenClaw for non-exec behavior unless further containment is added.

Option B: Build custom Phoenix node runtime
1. Stronger control over exec enforcement.
2. Higher long-term protocol compatibility and maintenance burden.
3. Best used after installer and sandbox foundations are stable.

Option C: Full runtime containment (VM-level)
1. Strongest isolation model.
2. Highest complexity and operational cost.
3. Appropriate for high-assurance security target.

## Decision Guidance

Recommended near-term order:
1. Finish approval websocket/UI path.
2. Ship deterministic installer/runtime packaging.
3. Implement sandboxed exec helper.
4. Add proxy-only egress enforcement.
5. Reassess need for custom node runtime after measuring residual risk.

## Open Questions

1. What minimum security bar is required for first public installer release?
2. Is same-user token storage acceptable until Keychain migration, or should Keychain be mandatory before release?
3. Do we treat non-exec gateway egress lock-down as release-blocking or phase 2 hardening?
4. At what point does custom node runtime ROI exceed maintenance cost?

   389	/// NSHostingView that zeroes its safe-area insets (so a hidden titlebar above
   390	/// doesn't inflate SwiftUI's ideal size) and keeps its hosting window sized
   391	/// to the SwiftUI content's intrinsic height. We drive the window frame
   392	/// directly rather than going through `setContentSize`, because the latter
   393	/// treats the SwiftUI ideal as *content-area* size and then adds the titlebar
   394	/// back on — producing a window that is taller than the content by exactly
   395	/// the titlebar height.
   396	final class SelfSizingHostingView<Content: View>: NSHostingView<Content> {
   397	    /// When true, intrinsic-driven resizes (after the initial sizing) animate
   398	    /// via AppKit's setFrame animator. The menu-bar panel keeps this off
   399	    /// because it has its own fade orchestration; the installer window flips
   400	    /// it on so screen transitions resize smoothly instead of snapping.
   401	    var animatesWindowResize: Bool = false
   402	
   403	    /// When true, the post-resize frame is clamped to the host screen's
   404	    /// `visibleFrame` so growth (which pins the top edge and pushes the
   405	    /// bottom down) cannot slide the window under the Dock or above the
   406	    /// menu bar. The menu-bar panel keeps this off because it's anchored
   407	    /// to the menu bar by construction; floating windows that the user
   408	    /// repositions (the installer is `window.center()`-ed at open) need
   409	    /// the clamp because top-edge pinning otherwise compounds with growth
   410	    /// to march the window off-screen on tall content / small displays.
   411	    var clampsToScreen: Bool = false
   412	
   413	    override var safeAreaInsets: NSEdgeInsets { NSEdgeInsets() }
   414	
   415	    override func invalidateIntrinsicContentSize() {
   416	        super.invalidateIntrinsicContentSize()
   417	        syncWindowSize(animate: animatesWindowResize)
   418	    }
   419	
   420	    override func viewDidMoveToWindow() {
   421	        super.viewDidMoveToWindow()
   422	        // Initial sizing (window just opened) should never animate — there's
   423	        // no prior frame to animate from.
   424	        syncWindowSize(animate: false)
   425	    }
   426	
   427	    private func syncWindowSize(animate: Bool) {
   428	        guard let window else { return }
   429	        let size = intrinsicContentSize
   430	        guard size.width > 0, size.height > 0 else { return }
   431	        if window.frame.size == size { return }
   432	        // Pin the top edge — both the menu-bar panel and the installer window
   433	        // anchor visually at the top, so growth must expand downward.
   434	        var frame = window.frame
   435	        frame.origin.y += frame.size.height - size.height
   436	        frame.size = size
   437	        if clampsToScreen, let visibleFrame = (window.screen ?? NSScreen.main)?.visibleFrame {
   438	            // Fix the top edge first, then the bottom — order matters so
   439	            // the bottom adjustment wins when the window is taller than
   440	            // the visible frame (degenerate; cap is supposed to prevent
   441	            // it). Result in that case: top pinned to visibleFrame.minY,
   442	            // bottom may extend above visibleFrame.maxY.
   443	            if frame.maxY > visibleFrame.maxY {
   444	                frame.origin.y = visibleFrame.maxY - frame.height
   445	            }
   446	            if frame.minY < visibleFrame.minY {
   447	                frame.origin.y = visibleFrame.minY
   448	            }
   449	        }
   450	        window.setFrame(frame, display: true, animate: animate)
   451	    }

### Finding 1 — medium
The height cap and frame clamp use different screen authorities. `InstallerView` computes the max height from `NSScreen.main`, but `SelfSizingHostingView` clamps position against `(window.screen ?? NSScreen.main).visibleFrame`. On a Mac where the installer is moved to a smaller non-main display, the SwiftUI frame can still be sized for the larger main display; the AppKit clamp only moves the oversized frame and cannot shrink it, so the intended ScrollView fallback does not re-engage and the installer can remain clipped/off-screen. Fix cost should stay low: make the sizing cap come from the same host-window screen source, not a new display heuristic layer.
