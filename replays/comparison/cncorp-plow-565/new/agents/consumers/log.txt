   389	/// NSHostingView that zeroes its safe-area insets (so a hidden titlebar above
   390	/// doesn't inflate SwiftUI's ideal size) and keeps its hosting window sized
   391	/// to the SwiftUI content's intrinsic height. We drive the window frame
   392	/// directly rather than going through `setContentSize`, because the latter
   393	/// treats the SwiftUI ideal as *content-area* size and then adds the titlebar
   394	/// back on — producing a window that is taller than the content by exactly
   395	/// the titlebar height.
   396	final class SelfSizingHostingView<Content: View>: NSHostingView<Content> {
   397	    /// When true, intrinsic-driven resizes (after the initial sizing) animate
   398	    /// via AppKit's setFrame animator. The menu-bar panel keeps this off
   399	    /// because it has its own fade orchestration; the installer window flips
   400	    /// it on so screen transitions resize smoothly instead of snapping.
   401	    var animatesWindowResize: Bool = false
   402	
   403	    /// When true, the post-resize frame is clamped to the host screen's
   404	    /// `visibleFrame` so growth (which pins the top edge and pushes the
   405	    /// bottom down) cannot slide the window under the Dock or above the
   406	    /// menu bar. The menu-bar panel keeps this off because it's anchored
   407	    /// to the menu bar by construction; floating windows that the user
   408	    /// repositions (the installer is `window.center()`-ed at open) need
   409	    /// the clamp because top-edge pinning otherwise compounds with growth
   410	    /// to march the window off-screen on tall content / small displays.
   411	    var clampsToScreen: Bool = false
   412	
   413	    override var safeAreaInsets: NSEdgeInsets { NSEdgeInsets() }
   414	
   415	    override func invalidateIntrinsicContentSize() {
   416	        super.invalidateIntrinsicContentSize()
   417	        syncWindowSize(animate: animatesWindowResize)
   418	    }
   419	
   420	    override func viewDidMoveToWindow() {
   421	        super.viewDidMoveToWindow()
   422	        // Initial sizing (window just opened) should never animate — there's
   423	        // no prior frame to animate from.
   424	        syncWindowSize(animate: false)
   425	    }
   426	
   427	    private func syncWindowSize(animate: Bool) {
   428	        guard let window else { return }
   429	        let size = intrinsicContentSize
   430	        guard size.width > 0, size.height > 0 else { return }
   431	        if window.frame.size == size { return }
   432	        // Pin the top edge — both the menu-bar panel and the installer window
   433	        // anchor visually at the top, so growth must expand downward.
   434	        var frame = window.frame
   435	        frame.origin.y += frame.size.height - size.height
   436	        frame.size = size
   437	        if clampsToScreen, let visibleFrame = (window.screen ?? NSScreen.main)?.visibleFrame {
   438	            // Fix the top edge first, then the bottom — order matters so
   439	            // the bottom adjustment wins when the window is taller than
   440	            // the visible frame (degenerate; cap is supposed to prevent
   441	            // it). Result in that case: top pinned to visibleFrame.minY,
   442	            // bottom may extend above visibleFrame.maxY.
   443	            if frame.maxY > visibleFrame.maxY {
   444	                frame.origin.y = visibleFrame.maxY - frame.height
   445	            }
   446	            if frame.minY < visibleFrame.minY {
   447	                frame.origin.y = visibleFrame.minY
   448	            }
   449	        }
   450	        window.setFrame(frame, display: true, animate: animate)
   451	    }

(connectors) — a magic number that didn't track when SettingsView's
content grew (more connectors, more containers, taller DownloadBarView
in the retry state). The result: a forced ScrollView at default content
size and bottom rows hidden behind the bottom-anchored DownloadBarView
during warm-up.

Switch the connectors screen to size dynamically from its measured
intrinsic content height + the overlay bar's measured height, capped to
the visible screen so the existing ScrollView still re-engages on
genuine overflow. Splash screens (welcomeInstall / FDA / activation /
connectContext) keep the design height of 530 because they fill via
Spacers rather than intrinsic content.

The previous attempt (PR #563, reverted in c06d059e + ad4e121f) used
the same PreferenceKey measurement pipeline but drove the window
imperatively via `window.animator().setFrame(...)` keyed off
NSApp.windows lookup. That fought AppKit's intrinsic-size re-pinning
on the SwiftUI hosting view (the new frame would be reverted on the
next layout pass) and was vulnerable to a startup race where the first
preference fired before `window.identifier = "installer"` was set, so
the window lookup returned nil and the resize was silently dropped.

This change reuses the SelfSizingHostingView pattern that already
backs MenuBarPanel: the SwiftUI tree pins its height via
`.frame(width: 680, height: installerWindowHeight(...))`, AppKit
treats that as intrinsicContentSize, and SelfSizingHostingView syncs
the window frame on every `invalidateIntrinsicContentSize`. AppKit
becomes the cooperator instead of the adversary, and the window
always exists by the time the hosting view is alive — both prior
failure modes are eliminated by construction.

SelfSizingHostingView is promoted from file-private (in
MenuBarController.swift) to internal so the installer can reuse it,
plus a new `animatesWindowResize` flag (default false, preserves the
menu-bar panel's snap behavior) so installer screen transitions
animate via AppKit's setFrame animator.

`installerWindowHeight(for:contentHeight:overlayHeight:cap:)` is a
pure file-scope function for unit testability — covered by
InstallerWindowSizeTests with synthetic inputs across splash,
connectors-with-content, connectors-with-overlay (incl. retry-state
growth), connectors-over-cap, and connectors-without-measurement
fallback.

`SelfSizingHostingView.syncWindowSize` clamp was reading
`window.screen ?? NSScreen.main`. Two screen sources → two visibleFrame
references → on a smaller secondary/Sidecar display (or after the user
drags the installer to one) the proposed height could exceed the host
display's visibleFrame. The clamp only repositions origin.y, never
shrinks size.height, so the bottom of the connectors / retry UI would
still slide off-screen.

Picked Option A (per the fix brief): plumb the host window's screen
into the SwiftUI tree so the cap reads from the same display the clamp
consults. `installerHeightCap` becomes a pure file-scope function
taking `visibleFrameHeight: CGFloat?`. `InstallerView` tracks the host
screen as `@State`, populated on `.onAppear` from the installer
window's `.screen` and refreshed on `NSWindow.didChangeScreenNotifi-
cation` (filtered by the "installer" identifier). The
SelfSizingHostingView clamp keeps reading `window.screen` — same
source, no second computation, no main-screen special case. End
result: the proposed height never exceeds the host visibleFrame in
the first place, so the clamp only ever has to reposition.

Option A was preferred over Option B (have the clamp shrink height to
fit) because A keeps the invariant in one place — InstallerView owns
"target height for the host display"; SelfSizingHostingView owns "put
the frame on-screen". B would have left two computations of the same
invariant (cap and clamp) and created the possibility of them
disagreeing again on the next refactor.

Tests: pulled `installerHeightCap` out as a pure helper so it's
directly unit-testable. Added four cases — three for the helper
itself (host-screen sourcing, 420pt floor at degenerate sizes, nil
fallback) and one end-to-end (`testSecondaryDisplayCapPreventsOver-
size`) reproducing the head-chef-flagged scenario: 600pt secondary
display visibleFrame + 800pt content + 100pt overlay → height capped
at 520pt (well inside the 600pt host visibleFrame).

screens, not when the host display's geometry changes in place
(resolution flip, display unplug, Sidecar attach/detach, menu-bar/Dock

visibleFrame; the round-2 observer stayed silent, the cap stayed
frozen at the pre-change visibleFrame, and the next syncWindowSize()
could leave the installer taller than the now-shrunken visible area —
re-opening the same secondary-display oversize bug round 2 was
supposed to close.

CGFloat? @State directly instead of an NSScreen?. The NSScreen-as-
identity model can't distinguish "same screen, different visibleFrame";
storing the CGFloat does, so SwiftUI sees a real diff on
didChangeScreenParametersNotification and re-evaluates the cap. The
prior `hostScreen` state had no consumer beyond reading
visibleFrame.height anyway, so dropping it removes a degree of
indirection rather than adding state. Subscribed to both notifications:
NSWindow.didChangeScreenNotification (window-crosses-screen, identifier-
filtered to "installer") and NSApplication.didChange-
ScreenParametersNotification (in-place display reconfiguration).
Refresh helper centralizes the `(window.screen ?? NSScreen.main)?.
visibleFrame.height` lookup so both observers + .onAppear stay in
lockstep.

Skipped the codex Minor's helper-extraction stretch goal: the would-be
helper boils down to a single ?? coalesce on two CGFloat? values, not
worth a function or a unit test. Added an inline comment at the

future reader doesn't simplify it back to one).

reverted (or kept under #if DEBUG) once the root cause is fixed.

Tag: every log line is prefixed `[PLO-35-DEBUG]` so the head chef can
`log stream | grep '[PLO-35-DEBUG]'` for a clean signal.

Instrumentation points:
- `refreshVisibleFrameHeight(from:)` — source notification (onAppear /
  didChangeScreen / didChangeScreenParameters), window lookup result,
  resolved screen vs NSScreen.main pointer (so we can see if we ever
  fall back to the wrong screen), the screen's full visibleFrame, and
  old vs new visibleFrameHeight value.
- `.onPreferenceChange(InstallerContentHeightKey)` — old + new + delta
  + whether the 1pt deadband filtered the update. Catches the failure
  mode where GeometryReader-inside-ScrollView reports 0 forever.
- `.onPreferenceChange(InstallerOverlayHeightKey)` — same shape, for
  the DownloadBarView measurement.
- `installerHeightCap(visibleFrameHeight:)` — input + output every
  call. (Will be noisy under SwiftUI body churn, intentionally so —
  catches the degenerate-cap failure mode.)
- `installerWindowHeight(...)` — all four inputs + the unclamped value
  + the final result every call. This is the actual height being fed
  into `.frame(height:)`.
- `SelfSizingHostingView.viewDidMoveToWindow` (installer-only) —
  initial intrinsicContentSize + window.frame.size at attach time.
  Confirms whether the SwiftUI tree's intrinsic ever reaches the
  hosting view at the right value.
- `SelfSizingHostingView.invalidateIntrinsicContentSize` (installer-
  only) — fires every time SwiftUI signals a new intrinsic. If we see
  windowHeight returning 800 but invalidateIntrinsic never firing
  with intrinsic.height=800, NSHostingView is dropping the update.
- `SelfSizingHostingView.syncWindowSize` (installer-only) — entry,
  intrinsic vs current frame, the early skip-if-already-matches path,
  pre-clamp / post-clamp frames with the screen identity used for the
  visibleFrame, what we asked setFrame for, and what window.frame
  reads immediately after (note: with animate: true the animator is
  still running, so the immediate read may not equal the final).

Coverage gap by design — SettingsView's GeometryReader closure is NOT
instrumented; if pref.content keeps firing 0 we'll know to add a log

The full pipeline downstream of the measurement is healthy: cap and
clamp share `0xbb2c7c780` (window.screen == NSScreen.main, the
secondary-display fix is working), SSHV.invalidateIntrinsic sees
intrinsic=680x530 and syncs cleanly. The window stays at 530 because
that's what installerWindowHeight is asking for — and it's asking for
530 because measuredContentHeight is stuck at 0.

Root cause: SwiftUI ScrollView on macOS 26 (Tahoe) lays out its content
lazily. SettingsView's
`.background(GeometryReader { Color.clear.preference(...) })` on the
inner padded VStack inside ScrollView fires once with
`geo.size.height = 0` BEFORE layout settles, the value dedupes against
the @State default (also 0), and SwiftUI never re-fires
.onPreferenceChange because the preference value never changes.
measuredContentHeight is therefore frozen at 0, installerWindowHeight
hits its connectors-without-measurement splash fallback (530), the
window stays at 530, ScrollView shows the scrollbar (acceptance #1),
and DownloadBarView overlaps content during warm-up (acceptance #2).

This is exactly failure mode #1 the original PR #563 brief warned
about: "GeometryReader inside `.background` of a ScrollView reports 0
on initial layout pass; preference never fires." Three rounds of pure-
helper unit tests passed because they fed synthetic non-zero
contentHeight to installerWindowHeight directly — they exercised the
math, not the SwiftUI plumbing.

Fix: add `.fixedSize(horizontal: false, vertical: true)` to the
padded-VStack BEFORE the .background measurement. .fixedSize forces
SwiftUI to use the view's intrinsic vertical size eagerly during the
first layout pass instead of waiting for ScrollView to settle, so the
.background's GeometryReader sees the real size on the first fire and
.onPreferenceChange propagates the non-zero value upstream. Once
measuredContentHeight reflects actual content, installerWindowHeight
sizes the window to fit, ScrollView's offered height matches the
content, the scrollbar disappears, and the existing scroll fallback
re-engages only when content > cap (the documented behavior).

Also added a `[PLO-35-DEBUG] settings.GR geo.size=...` NSLog inside the
GeometryReader closure so the next runtime capture confirms the
GeometryReader is firing with non-zero values. If `.fixedSize` doesn't
flip pref.content to non-zero, that log tells us exactly what
geo.size IS, which informs the next attempt.

wiped state): windowHeight returns 530 unchanged, intrinsic=680x530,
