The installer window's height was hard-coded to 530 (splash) / 610
(connectors) — a magic number that didn't track when SettingsView's
content grew (more connectors, more containers, taller DownloadBarView
in the retry state). The result: a forced ScrollView at default content
size and bottom rows hidden behind the bottom-anchored DownloadBarView
during warm-up.

Switch the connectors screen to size dynamically from its measured
intrinsic content height + the overlay bar's measured height, capped to
the visible screen so the existing ScrollView still re-engages on
genuine overflow. Splash screens (welcomeInstall / FDA / activation /
connectContext) keep the design height of 530 because they fill via
Spacers rather than intrinsic content.

The previous attempt (PR #563, reverted in c06d059e + ad4e121f) used
the same PreferenceKey measurement pipeline but drove the window
imperatively via `window.animator().setFrame(...)` keyed off
NSApp.windows lookup. That fought AppKit's intrinsic-size re-pinning
on the SwiftUI hosting view (the new frame would be reverted on the
next layout pass) and was vulnerable to a startup race where the first
preference fired before `window.identifier = "installer"` was set, so
the window lookup returned nil and the resize was silently dropped.

This change reuses the SelfSizingHostingView pattern that already
backs MenuBarPanel: the SwiftUI tree pins its height via
`.frame(width: 680, height: installerWindowHeight(...))`, AppKit
treats that as intrinsicContentSize, and SelfSizingHostingView syncs
the window frame on every `invalidateIntrinsicContentSize`. AppKit
becomes the cooperator instead of the adversary, and the window
always exists by the time the hosting view is alive — both prior
failure modes are eliminated by construction.

SelfSizingHostingView is promoted from file-private (in
MenuBarController.swift) to internal so the installer can reuse it,
plus a new `animatesWindowResize` flag (default false, preserves the
menu-bar panel's snap behavior) so installer screen transitions
animate via AppKit's setFrame animator.

`installerWindowHeight(for:contentHeight:overlayHeight:cap:)` is a
pure file-scope function for unit testability — covered by
InstallerWindowSizeTests with synthetic inputs across splash,
connectors-with-content, connectors-with-overlay (incl. retry-state
growth), connectors-over-cap, and connectors-without-measurement
fallback.

No more speculative patches. This commit adds DEBUG-gated NSLog
instrumentation at every link in the height-sizing chain so the next
build + capture pins down which link is actually broken. To be
reverted (or kept under #if DEBUG) once the root cause is fixed.

Tag: every log line is prefixed `[PLO-35-DEBUG]` so the head chef can
`log stream | grep '[PLO-35-DEBUG]'` for a clean signal.

Instrumentation points:
- `refreshVisibleFrameHeight(from:)` — source notification (onAppear /
  didChangeScreen / didChangeScreenParameters), window lookup result,
  resolved screen vs NSScreen.main pointer (so we can see if we ever
  fall back to the wrong screen), the screen's full visibleFrame, and
  old vs new visibleFrameHeight value.
- `.onPreferenceChange(InstallerContentHeightKey)` — old + new + delta
  + whether the 1pt deadband filtered the update. Catches the failure
  mode where GeometryReader-inside-ScrollView reports 0 forever.
- `.onPreferenceChange(InstallerOverlayHeightKey)` — same shape, for
  the DownloadBarView measurement.
- `installerHeightCap(visibleFrameHeight:)` — input + output every
  call. (Will be noisy under SwiftUI body churn, intentionally so —
  catches the degenerate-cap failure mode.)
- `installerWindowHeight(...)` — all four inputs + the unclamped value
  + the final result every call. This is the actual height being fed
  into `.frame(height:)`.
- `SelfSizingHostingView.viewDidMoveToWindow` (installer-only) —
  initial intrinsicContentSize + window.frame.size at attach time.
  Confirms whether the SwiftUI tree's intrinsic ever reaches the
  hosting view at the right value.
- `SelfSizingHostingView.invalidateIntrinsicContentSize` (installer-
  only) — fires every time SwiftUI signals a new intrinsic. If we see
  windowHeight returning 800 but invalidateIntrinsic never firing
  with intrinsic.height=800, NSHostingView is dropping the update.
- `SelfSizingHostingView.syncWindowSize` (installer-only) — entry,
  intrinsic vs current frame, the early skip-if-already-matches path,
  pre-clamp / post-clamp frames with the screen identity used for the
  visibleFrame, what we asked setFrame for, and what window.frame
  reads immediately after (note: with animate: true the animator is
  still running, so the immediate read may not equal the final).

Coverage gap by design — SettingsView's GeometryReader closure is NOT
instrumented; if pref.content keeps firing 0 we'll know to add a log
inside the GeometryReader body next round.
