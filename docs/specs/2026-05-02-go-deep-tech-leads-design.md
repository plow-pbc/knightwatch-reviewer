# Go-Deep Tech Leads — Design

## Problem

The Broken-Glass Reviewer work merged in 2026-05-01 changed the *voice posture* of kw-reviewer's findings (questions over prescriptions, cost-naming for additive remedies) but didn't change the *upstream signal*: specialists still find every drift class they can, regardless of whether the proposed remedy is worth its complexity at the repo's actual scale. Subsequent live observation across `cncorp/plow#546` (9 rounds, declined) and `cncorp/plow#575` (BCR finding for a sub-second auth race at 10 users) shows three structural failure modes:

1. **Specialists keep finding edge-case drift the operator has already declined.** PR #546 had the same session-scoping finding flagged 9 rounds in a row; the operator declined each round citing documented design intent (`testFinishKeepsLaunchPhaseLaunching`); the bot re-flagged from scratch each round. There is no decline-history mechanism — kw-reviewer is stateless across rounds.
2. **The bot's structural recommendations are net-additive without scale calibration.** PR #575's BCR finding ("single-owner auth-state contract") is correct architecturally but proposes ~150-300 LOC of refactor for a race that fires <1× per user-month at ~10 users. The bot identifies the simpler architecture but doesn't ask "is this worth doing now at our scale?"
3. **No dedicated codebase-pattern search.** When a specialist proposes "add X," there's no agent whose primary job is "is there an existing pattern Y in the codebase we could reuse to delete that proposal?" The `simplification` specialist covers prior-art DRY; nothing covers "given the proposed remedy, what's the simplest alternative shape?"

The 2026-05-01 broken-glass work changed the *output* — same findings, more inquisitive voice. This spec changes the *input pipeline*: specialists keep doing what they do, but a new fan-out stage challenges high-LOC findings with deep codebase research, and the critic gains decline-history awareness so the operator's prior pushback isn't ignored.

## The cultural lens this spec encodes

Two questions every high-LOC finding must answer before being raised as `medium` or `blocking`:

1. **Is the additional complexity of addressing this issue worth the cost of slowing down PMF iteration?** Use the per-repo `.knightwatch/review-priority.md` (now committed in 5 product repos) for the operating-point anchor.
2. **Is there a simpler architecture that delivers the same user intent (cite the `Inferred intent:` line)?** Search the codebase. If yes, propose that as the alternative rather than adding new code.

These ladder into the broken-glass posture from 2026-05-01: questions over prescriptions, cost-naming for additive remedies. This spec operationalizes them as actual investigative steps in the review pipeline rather than prompt-level admonitions.

## What's already in place

- `Broken-Glass Test` (vibe-engineering `CODING_STANDARDS.md`) — universal voice posture + question template + cost-naming clause.
- `prompts/critic.md` — has `REMEDY-BLOAT` / `REFRAME-AS-QUESTION` status buckets.
- `prompts/aggregator.md` — has structured `Open Questions` section with `**Q:**` template.
- `prompts/momentum.md` — re-review meta-finding specialist, runs only on re-reviews.
- `lib/loc-trend.sh` — `compute_loc_trend()` + typed row states (`unavailable`, `reachable_zero`, `deletion_only`, `numeric`).
- `lib/run-dir.sh` — `is_run_author_visible`, `author_visible_runs_iter`, `latest_author_visible_review_*` helper family.
- `prior-reviews.md` scratch input — concatenated prior aggregator outputs.
- Per-repo `.knightwatch/review-priority.md` files merged in 5 product repos (PRs `cncorp/plow#577`, `cncorp/plow-content#4`, `srosro/tkmx-client#20`, `srosro/tkmx-server#22`, `plow-pbc/watchmepivot#9`). These are PREREQUISITES for this spec; the bot needs the per-repo operating point to do the calibration.

## Goals

1. **Specialist-file co-location.** All review data about a finding (the specialist's framing + critic's counter-arg/calibration questions + go-deep tech-lead's investigation) lands in ONE file per specialist (`specialists/<angle>.md`), so the aggregator reads a single file per specialist with full context.
2. **Decline-history awareness.** The bot reads operator replies on prior reviews; if a finding-class has been declined N times, the next round drops the finding (or escalates to "previously declined" footnote rather than re-raising as blocking).
3. **20-LOC remedy threshold.** Findings with proposed remedies ≥20 LOC get systematic calibration questions generated by the critic + investigated deeply by a go-deep tech lead.
4. **Targeted depth, auto-scaling cost.** Up to 3 go-deep tech-leads fan out per review, one per "hot" specialist file (a specialist file containing at least one top-X finding). Simple PRs with low-LOC remedies → 0 go-deep runs. Complex PRs → up to 3 in parallel.
5. **Codebase-pattern search as a first-class lens.** Each go-deep tech-lead has full repo access (grep, file reads, git log, kid prior-art) and is mandated to find existing infrastructure that could simplify the finding's remedy.
6. **Engagement with documented design intent.** When proposing a structural change, go-deep tech-leads grep tests/ for assertions documenting the design choice and engage with them rather than re-arguing in the abstract.
7. **Reduced bot complexity through clearer responsibilities.** Critic gets a sharper mandate (stress-test + calibration question generation); go-deep tech-leads own the deep investigation. The architecture is more modular even though kw-reviewer's LOC grows.

## Non-goals

- No replacement of any existing specialist. The 8 finding-producing specialists keep their mandates.
- No change to `learn-from-replies.sh` / `/srosro-memorize`. Future tuning still flows through that seam.
- No change to first-review behavior (no momentum, no decline-history — these are re-review-only signals).
- No new orchestrator-level state. All new data flows via the existing `.codex-scratch/` scratch-file pattern + the `runs/` metadata seam.
- No "real-bug pre-pass" that filters specialists before fan-out. Considered earlier and deferred — premature optimization.

## Architecture

The pipeline gains TWO new stages (decline-history fetch + go-deep tech-lead fan-out) and FOLDS the critic's output into the specialist files via post-processing:

```
Stage 0 (NEW): orchestrator fetches operator replies → decline-history.md
   ↓
Stage 1 (UNCHANGED): specialists fan out (8 parallel) → specialists/<angle>.md
   ↓
Stage 2 (EXTENDED): critic (single) → critic.md
   • counterarguments per specialist finding (existing REMEDY-BLOAT et al.)
   • per-finding remedy-LOC estimate (NEW)
   • for ≥20 LOC findings: generate 1-2 calibration questions (NEW)
   • reads decline-history.md; drop or footnote findings the operator has declined ≥3 times (NEW)
   ↓
Stage 2.5 (NEW orchestrator post-processing): split critic.md by [<angle>] sections,
   APPEND each section to specialists/<angle>.md
   ↓
Stage 3 (NEW orchestrator step): rank top X (≤3) findings overall; group by specialist file → "hot" specialists
   ↓
Stage 4 (NEW): go-deep tech-leads fan out (≤3 parallel — one per hot specialist file)
   • each receives: the augmented specialist file + full repo access + .knightwatch/* + decline-history.md
   • mandate: rank complexity vs benefit; find alternatives that satisfy inferred-intent more simply
   • output: appends to its assigned specialist file
   ↓
Stage 5 (LIGHTLY EXTENDED): aggregator reads specialists/<angle>.md (now 3-layered) → final review
```

**Specialist file structure (after all stages):**

```markdown
## [<angle>] findings

### Surveyed
- ...
### Finding 1 — <severity>
...

---

## Critic counter-arguments

### Finding 1 — <verdict>
counter-arg / calibration questions / decline-history note (if applicable)

### Finding 2 — REFRAME-AS-QUESTION
...

---

## Go-deep tech-lead investigation

### Investigation of Finding 1
**Calibration:**
- Q (from critic): "Will users hit this state at ~10 users?"
  - A: ~0 firing rate observed; not a pattern in production logs.
- Q (from critic): "Is there a similar pattern in the codebase we can reuse?"
  - A: yes — `lib/auth.sh:42` already has the single-owner shape; cite.

**Recommendation:** SIMPLIFY-WITH-PATTERN — propose using `lib/auth.sh:42` instead of new code (-30 LOC vs current diff). Alternatively, REFRAME-AS-QUESTION since the firing rate is unobserved.
```

**Invariant: every specialist file has at most one writer at any moment.**

| Stage | Writer per file |
|---|---|
| 1 | the specialist itself (parallel, but each to its own file) |
| 2.5 | orchestrator (sequential per-file appends from critic.md) |
| 4 | the go-deep tech-lead (parallel, but max 1 per file — at most 3 total) |
| 5 | aggregator reads only |

No race conditions; no flock needed; data co-located in one file per specialist.

## Phasing

### Phase 1 — foundation (decline-history + critic generates questions + co-location)

Captures ~70% of the value; no new fan-out infrastructure.

**Changes:**

#### Phase 1, Change A — `decline-history.md` scratch input

New orchestrator step before specialist fan-out. Fetches operator comments via `gh pr view --json comments` + `gh api .../pulls/<#>/comments` (inline). Parses for reply patterns matching the babysit-pr decline templates (`Declined — conflicts with`, `Counter-proposed`, etc.) and the operator's free-form pushback.

Output format (`.codex-scratch/decline-history.md`):

```markdown
# Decline history

Operator (srosro) has declined the following finding classes on prior reviews of this PR:

## Class: session-scoping (declined 9 rounds)
- Last decline reason: "session lifetime ≠ launch-attempt lifetime; documented in tests/test_runtime.swift:testFinishKeepsLaunchPhaseLaunching"
- First flagged: round 1 (commit X)
- Last flagged: round 9 (commit Y)

## Class: stale-auth-error (declined 1 round)
- Last decline reason: "edge-case for 10 users; deferred — see follow-up issue #N"
```

The class identifiers (`session-scoping`, `stale-auth-error`) are extracted by the parser from the bot's prior `[Bug-Class-Recurrence]` framings + the babysit-pr Counter-propose triage tags.

**Implementation:** new helper `lib/decline-history.sh` with `fetch_decline_history <repo> <pr_num> <state_dir>`. Called from `lib/review-one-pr.sh` before specialist fan-out, output written to `.codex-scratch/decline-history.md`. ~80 LOC.

**Smoke test:** `lib/tests/decline-history-smoke.sh` — fixture with synthetic PR comments, verify class extraction + decline-count math. ~50 LOC.

#### Phase 1, Change B — Critic reads decline-history; drops or footnotes recurring findings

`prompts/critic.md` extended:

- New input: `.codex-scratch/decline-history.md`.
- Per surviving specialist finding, check whether its class matches any in decline-history.
- If declined ≥3 rounds: drop from the published findings; emit a one-line footnote *"Class 'X' has been declined N rounds; see decline-history.md. Not re-raising."*
- If declined 1-2 rounds: keep the finding but cite the operator's prior reasoning + ask whether the new commit affects the prior decline.

This is the PR #546 fix — would have collapsed 9 rounds → 1.

**Implementation:** ~30 LOC additions to `prompts/critic.md`.

#### Phase 1, Change C — Critic estimates remedy LOC + generates calibration questions for ≥20 LOC findings

For each surviving specialist finding, the critic estimates the LOC of the proposed remedy (rough: small enough not to need an LLM call's worth of effort — count `+` LOC in any code blocks the specialist proposed, fall back to "if the finding cites N files, estimate N×20 LOC").

For findings ≥20 LOC remedy:
- Generate 1-2 calibration questions per finding. Examples (LLM generates per-finding, not templated):
  - *"Will users at ~10 users hit this state? Cite firing-rate evidence if available."*
  - *"Is there a similar pattern in `lib/X.sh` we could reuse to avoid adding N LOC?"*
  - *"Does the operator's prior decline (if any) still apply to this commit's diff?"*
- The questions ladder up to: *"Is the additional complexity of addressing this issue worth the cost of slowing down PMF iteration?"*

Output format (in critic.md, alongside the existing per-finding sections):

```markdown
### [<angle>] Finding N — <verdict: AGREE | REFRAME-AS-QUESTION | REMEDY-BLOAT | ...>
counter-arg or calibration setup

**Estimated remedy LOC:** ~150 LOC across 3 files.

**Calibration questions for go-deep investigation:**
- Q1: Will the auth race fire at ~10 users? Look for similar races in production logs / observed failures.
- Q2: Is there a single-owner pattern in `lib/auth.sh` or `lib/state-io.sh` we can reuse?
```

**Implementation:** ~50 LOC additions to `prompts/critic.md`.

#### Phase 1, Change D — Orchestrator splits critic.md and appends to specialist files

After critic completes, orchestrator parses `critic.md` for `### [<angle>] Finding N` sections, groups by `<angle>`, appends each group to `specialists/<angle>.md`. The specialist files now have BOTH the original specialist's findings AND the critic's per-finding counter-args + calibration questions co-located.

**Implementation:** ~30 LOC bash in `lib/review-one-pr.sh` (or a new `lib/critic-splitter.sh` helper).

**Smoke test:** `lib/tests/critic-splitter-smoke.sh` — fixture with synthetic critic.md + specialist files; verify the split + append produces correctly augmented files. ~40 LOC.

#### Phase 1, Change E — § Broken-Glass Test gains 20-LOC remedy threshold

vibe-engineering `claude-config/CODING_STANDARDS.md § Broken-Glass Test` gets a new sub-rule:

```markdown
### 20-LOC remedy threshold

When a proposed remedy adds more than ~20 LOC, the reviewer must explicitly answer three questions before raising the finding as `medium` or higher:

1. **Will this affect our core product and help us find PMF?** Or is this covering an edge case?
2. **If covering an edge case:** is there a simpler contract or infrastructure (a single-shape, fail-loud seam) that catches this AND other future edge cases for free? If yes, propose the contract instead of the local handler.
3. **If neither (1) nor (2):** is the complexity worth it, given (a) ~10 users won't experience the edge case at our current scale, and (b) the LOC cost slows down PMF iteration?

Default when (1)+(2)+(3) all fail: **ship simple-and-fail-loud**, not elegant-defensive-coverage. We want simple, elegant code with some broken glass that fails loudly — the cost of a calcified branch outweighs the cost of a loud crash the operator can investigate.
```

**Implementation:** ~25 LOC addition to `CODING_STANDARDS.md` in vibe-engineering.

### Phase 2 — go-deep tech-leads (depth, fan-out)

Adds the dedicated investigator specialists. Builds on Phase 1's calibration-question generation.

**Changes:**

#### Phase 2, Change F — `prompts/go-deep.md` (new specialist)

Single prompt, instantiated up to 3× per review (one per hot specialist file). Each instance receives a different augmented specialist file as input.

**Mandate:**
- Read the assigned specialist file (already augmented by Phase 1's critic-splitter with counter-args + calibration questions).
- For each finding with calibration questions, INVESTIGATE deeply:
  - Use `grep -rn` across the repo for similar patterns.
  - Use `git log --grep=...` for history (was this added before? was it reverted?).
  - Use kid prior-art for cross-repo patterns.
  - Read `tests/` for assertions documenting the design choice.
  - Read `.knightwatch/review-priority.md` for the repo's operating point.
  - Read `decline-history.md` for prior pushback.
- For each calibration question: produce an answer with `path:line` evidence + confidence.
- Per finding: emit a recommendation (KEEP / SIMPLIFY-WITH-PATTERN / DROP / REFRAME).

**Output format** (appended to the specialist file under a new `## Go-deep tech-lead investigation` section):

```markdown
---

## Go-deep tech-lead investigation

### Investigation of Finding 1

**Calibration answers:**

**Q1: Will the auth race fire at ~10 users?**
A: Searched git log for `auth-error-file` writes — last 90 days show 0 instances of the race firing in production logs. No related issue on the tracker. Confidence: high.
Evidence: `git log -p --since='90 days ago' -- 'app/openclaw/channels/plow-imessage/src/send.ts'` shows 7 commits to send.ts, none mention the race.

**Q2: Is there a single-owner pattern in `lib/auth.sh` we can reuse?**
A: Partial — `app/plowd/plowd/main.py:1175` already owns `auth-profiles.json` writes. The race exists because `app/openclaw/channels/plow-imessage/src/send.ts:92` ALSO writes auth-error-file via a different code path. Single-owner refactor would remove the second writer.
Evidence: `grep -rn "auth-profiles.json" app/` shows 2 writers (plowd:1175 + send.ts:92).

**Recommendation:** REFRAME-AS-QUESTION
- Race is real but firing rate is observably zero in production at current scale.
- Simpler architecture exists (remove send.ts as a writer), cost ~80 LOC.
- At 10 users, the race's user impact (30s confusion, fix-by-re-OAuth) is recoverable.
- Recommend deferring; revisit if firing rate observed in production.
```

**Implementation:** ~80 LOC for the prompt.

#### Phase 2, Change G — Orchestrator: ranker + go-deep fan-out

Two new orchestrator steps, after Phase 1's critic-splitter:

**G.1 Ranker step** (new in `lib/review-one-pr.sh`, ~50 LOC):
- Read augmented `specialists/<angle>.md` files.
- Parse for findings with calibration questions (Phase 1 wrote these).
- Rank top X (≤3) findings overall by:
  - Severity (`blocking` > `medium` > `low` > `nit`).
  - Critic verdict filter (skip findings the critic dropped via `REMEDY-BLOAT` / `FALSE POSITIVE`).
  - Tiebreak by remedy LOC, **descending** (biggest LOC first — go-deep targets the highest-cost findings, since ≥20 LOC is the population we want investigated).
- Group top findings by their specialist file → set of "hot" specialists.

**G.2 Go-deep fan-out** (new in `lib/review-one-pr.sh`, ~30 LOC):
- For each hot specialist (up to 3): invoke `run-specialist.sh` in parallel, passing the augmented specialist file as input.
- Each instance is identified by `<angle>` (e.g. `go-deep-security`) so its output dir is `RUN_DIR/agents/go-deep-<angle>/output.md` (avoids races between parallel instances), but **all instances use the same `prompts/go-deep.md` template**.
  - Implementation: extend `run-specialist.sh` with a `--prompt-name <name>` flag that decouples the output dir name from the prompt file path. Default behavior (no flag) keeps the existing 1:1 name↔prompt mapping. ~10 LOC of the +30 budgeted here goes to that flag.
- Wait for all go-deep PIDs (mirrors existing specialist fan-out wait logic).
- Orchestrator appends each go-deep output to its assigned `specialists/<angle>.md`.

**Smoke test:** `lib/tests/go-deep-fanout-smoke.sh` — token-level fence on the orchestrator wiring (rank step is invoked, fan-out happens, append step happens). ~50 LOC.

#### Phase 2, Change H — Aggregator reads layered specialist files

`prompts/aggregator.md` extended (lightly):

- Note that specialist files now have THREE layers (specialist + critic + go-deep tech-lead).
- For each finding, the aggregator integrates the go-deep recommendation:
  - `KEEP` → publish as-is (with severity from specialist + critic verdict).
  - `SIMPLIFY-WITH-PATTERN` → rewrite the finding's remedy to use the cited pattern; severity stays.
  - `DROP` → omit from published findings (footnote: "X was investigated; decline reason: …").
  - `REFRAME` → move to Open Questions with the go-deep's calibration answers as the question body.

**Implementation:** ~30 LOC additions to `prompts/aggregator.md`.

### Combined phases

Phase 1 + Phase 2 = **~+570 LOC in kw-reviewer + ~+25 LOC in vibe-engineering**.

That's substantial. The justification is in the bot OUTPUT — fewer over-iteration loops (PR #546 class), fewer net-additive findings on edge cases (PR #575 class), and explicit codebase-pattern search that turns "add 50 LOC" recommendations into "use existing 5-LOC helper at lib/X.sh:42." The downstream LOC savings (in PR diffs not produced) should be 2-3× the kw-reviewer LOC additions.

## Risks

- **Phase 1 critic prompt grows substantially.** Adding remedy-LOC estimation + calibration question generation + decline-history awareness layers significant cognitive load on a single prompt. Mitigation: explicit per-finding output structure; smoke fences each new contract token.
- **Phase 2 cost auto-scaling could miss-fire.** If the ranker logic picks "top 3 findings" too aggressively (e.g. 3 minor findings all at `medium` severity), three go-deep tech-leads run on small remedies. Mitigation: ranker filter requires the calibration questions to exist (i.e. critic flagged remedy as ≥20 LOC); if no findings meet the threshold, no go-deep runs. Auto-scales to 0 on simple PRs.
- **Decline-history parser could mis-classify.** The babysit-pr decline templates are stable but operator free-form replies are not. Mitigation: parser starts conservative (only matches known templates); operator can manually annotate with `<!-- decline:class=X -->` HTML comments to bridge cases the parser misses.
- **Go-deep prompt could over-search.** Each go-deep has full repo access + grep + git log; could spend tokens roaming. Mitigation: prompt mandates a structured output per calibration question (path:line + confidence + recommendation), forcing the agent to focus on producing the answer rather than exploring.
- **Critic-splitter contract drift between phases.** Phase 1 ships the critic-to-specialist-file split; Phase 2 assumes that data flow exists. If Phase 1's split has a bug (e.g. drops sections), Phase 2 breaks silently. Mitigation: explicit smoke (critic-splitter-smoke.sh) fences the split contract; Phase 2's go-deep-fanout-smoke depends on it.

## Open questions

- **Should go-deep tech-leads use `kid` prior-art?** Current `kid` index is per-repo (`KID_PATHS` map in `repos.conf`). Cross-repo similarity would be useful for "is there a pattern in another tracked repo we could reuse?" — but kid is currently scoped to within-repo per-repo. Possible future extension; out of scope for Phase 2.
- **Should the decline-history parser handle `/srosro-memorize` calibrations?** Currently `learn-from-replies.sh` auto-tunes `COMMENT_REVIEW_MISTAKES.md` from author replies. Decline-history could similarly tune; but this spec keeps decline-history *per-PR* (not global), so they're orthogonal. Worth revisiting after live observation.
- **Severity bumps based on go-deep recommendation?** If go-deep finds "the simpler shape would delete 100 LOC and the bug is real," should the severity be elevated? Current design says "no — go-deep recommends but doesn't bump severity." Could revisit if the bot keeps under-flagging.
- **What happens when there are 0 hot specialists (no findings ≥20 LOC remedy)?** Phase 2 ranker correctly produces an empty hot-list; no go-deep runs. Aggregator publishes findings as Phase 1 prepared them. Good auto-scale to small PRs.
